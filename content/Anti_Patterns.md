## 📝 TL;DR

Більшість проблем із Claude Code — не баги моделі, а передбачувані помилки у взаємодії з нею. Цей каталог описує десять найпоширеніших антипатернів: симптом, причину та конкретний спосіб уникнути.

## 1. "Просто зроби це" без планування

**Симптом:** Ви дали розмите завдання рівня фічі — агент пішов і через 20 хвилин виніс 400-рядковий diff, який ви не можете пояснити на code review.

**Чому це погано:** Без явного списку задач агент інтерпретує scope самостійно. Він "допомагає" рефакторити сусідні класи, перейменовує змінні і додає логування — все в одному коміті.

**Як уникнути:** Перед будь-якою нетривіальною задачею — `/plan`. Вийти з Plan Mode можна лише після того, як ви схвалили список кроків. Один крок = один коміт.

Детальніше: [Планування та Tasks API](Planning_and_Tasks.md), [Золотий workflow](Golden_Workflow.md)

---

## 2. Довіра без верифікації

**Симптом:** "Агент написав — значить правильно." Ви приймаєте зміни без `git diff`, без запуску тестів, без review. CI падає вже після merge.

**Чому це погано:** Claude впевнено генерує правдоподібний код, навіть коли помиляється. Особливо небезпечно у граничних випадках: транзакційні межі, порядок Flyway-міграцій, Kafka offset semantics.

**Як уникнути:** Замкніть [верифікаційний цикл](Agentic_Verification_Loop.md): кожен крок завершується зеленим тестом від [`test-runner`](Test_Runner_Subagent_Example.md), а фінальний diff — вердиктом [`reviewer`](Reviewer_Subagent_Example.md). Ніколи не мержте без незалежного погляду на зміни.

## 3. Розмитий промпт → зміни поза scope

**Симптом:** Ви просили "виправити баг у `PaymentService`" — агент виправив баг і заодно "покращив" `OrderService`, `UserMapper` та три тести, яких ви не просили чіпати.

**Чому це погано:** Агент не має поняття "scope" якщо ви його не задали явно. Він оптимізує "якість коду", а не виконання конкретного завдання.

**Як уникнути:**

- Вказуйте файли явно: *"Fix the bug in `PaymentService.java` only. Do not touch any other files."*
- Після кожного кроку: *"Implement Task N only. Stop and wait for confirmation."*
- Перед комітом перевіряйте `git diff --stat` — несподіваний файл у списку = сигнал тривоги.

Детальніше: [Ефективні промпти](Effective_Prompting.md)

## 4. Контекст-монстр

**Симптом:** Сесія триває 3 години. Агент починає ігнорувати правила з `CLAUDE.md`, повторює помилки, які вже виправляв, "забуває" домовленості з початку розмови.

**Чому це погано:** Це [ефект "Lost in the Middle"](Context_Window_Management.md) — LLM погано утримує інформацію з середини довгого контексту. Ранні правила та рішення поступово "тонуть".

**Як уникнути:**

- `/compact` на межах між логічними кроками з явними інструкціями що зберегти.
- `/clear` коли поточна сесія зайшла в глухий кут — простіше почати заново.
- Делегуйте "брудну" роботу (читання файлів, аналіз логів) субагентам — вони стартують зі свіжим контекстом.

## 5. Фікс тестів через їх видалення

**Симптом:** Тест падав. Після роботи агента — не падає. Ви раді. Через тиждень у production виявляєте баг, який цей тест і покривав.

**Чому це погано:** Якщо промпт звучить як *"fix the failing test"* — агент може видалити тест, закоментувати assertion або замінити `assertEquals` на `assertTrue(true)`. Технічно тест "проходить".

**Як уникнути:** Завжди формулюйте як *"fix the **code** so the failing test passes"*, а не *"fix the test"*. Після виправлення — перевірте, що тест досі перевіряє те, що мав перевіряти.

---

## 6. Відсутність checkpoint-стратегії

**Симптом:** Агент виконав 5 кроків підряд без єдиного коміту. На кроці 4 щось пішло не так, але відкотитись нема куди — `git diff` показує 800 змінених рядків.

**Чому це погано:** Без git-checkpoint кожен наступний крок збільшує "вартість відкату". Якщо агент зайшов у глухий кут — ви змушені або приймати всі зміни, або відкидати все.

**Як уникнути:** Після кожного верифікованого кроку: *"Commit this step with a conventional commit message."* Дрібні коміти — не педантизм, а страховка.

Детальніше: [Git та PR Workflow](Git_PR_Workflow.md)

---

## 7. Архітектурні рішення без `/think`

**Симптом:** Агент реалізував DLQ-обробник прямо всередині `@Transactional`-методу. Або Flyway-міграція V12 модифікує існуючу V11. Або новий Kafka consumer підписується на топік без dead-letter fallback.

**Чому це погано:** Для типових CRUD-задач Claude достатньо. Але для рішень з прихованими інваріантами (транзакційні межі, порядок міграцій, ordering гарантії Kafka) стандартний режим може генерувати правдоподібно-невірний код.

**Як уникнути:** Перед реалізацією крос-шарових або ризикових змін:

```text
/plan
think hard: [конкретне архітектурне питання з обмеженнями]
```

Детальніше: [Extended Thinking та /think](Extended_Thinking.md)

## 8. Scope Creep при виправленні

**Симптом:** Ви просили виправити один метод. Агент переписав клас "поки вже тут", змінив сигнатуру публічного API і витягнув три приватні методи в окремий helper. Нові тести не написав.

**Чому це погано:** Claude за замовчуванням схильний до "покращення" коду навколо задачі. Без явного обмеження scope кожне виправлення ризикує перетворитись на незапланований рефакторинг.

**Як уникнути:**

- Зафіксуйте обмеження в `CLAUDE.md`: *"Do not refactor code that is not directly related to the current task."*
- У промпті: *"Make the minimal change needed. Do not refactor surrounding code."*
- [`reviewer`](Reviewer_Subagent_Example.md) перед комітом виявляє незапитані зміни у вердикті.

---

## 9. Silent Failures — приховування помилок

**Симптом:** Тест проходить, але в логах немає очікуваних результатів. У production "зникають" події або транзакції — без жодного exception.

**Чому це погано:** Агент може додати `catch (Exception e) { log.warn("failed", e); }` замість `throw`, щоб прибрати червоний тест. Код компілюється, тест зеленіє — але помилка поглинається мовчки. Особливо небезпечно у Kafka listeners і `@Async`-методах, де exception у фоновому потоці не видно назовні.

**Як уникнути:**

- У промпті явно: *"Do not add empty catch blocks or swallow exceptions. Re-throw or propagate."*
- Після виправлення перевіряйте diff на наявність нових `try-catch` — це сигнал тривоги.
- [`reviewer`](Reviewer_Subagent_Example.md) має перевіряти "exception swallowing" як окремий критерій.

---

## 10. Нескінченний retry-loop

**Симптом:** Агент пробує виправити одну й ту саму помилку 5–10 разів поспіль, кожного разу роблячи незначні варіації. Токени горять, прогресу нема.

**Чому це погано:** Без обмеження кроків агент може зациклитись на локальній проблемі — особливо якщо помилка потребує архітектурного рішення, а не чергової правки рядка.

**Як уникнути:**

- У CLI: `--max-turns N` фізично обмежує кількість кроків (детальніше: [Headless-режим](Headless_Mode.md)).
- В інтерактивній сесії: якщо агент не розв'язав проблему за 3 спроби — зупиніть його, поверніться до `/plan` та переформулюйте задачу.
- Коміт після кожного кроку (антипатерн 6) дає точку відкату замість накопичення невдалих правок.

## Зведена таблиця

| Антипатерн | Сигнал тривоги | Швидке виправлення |
| :--------- | :------------- | :----------------- |
| Без планування | Diff > 200 рядків з першої спроби | `/plan` перед стартом |
| Довіра без верифікації | CI падає після merge | `test-runner` + `reviewer` на кожен крок |
| Розмитий scope | Несподівані файли у `git diff --stat` | Вказати файли явно у промпті |
| Контекст-монстр | Агент повторює вже виправлені помилки | `/compact` або `/clear` |
| Фікс тестів видаленням | Тест "пройшов" але assertion зник | "fix the code, not the test" |
| Без checkpoint | `git diff` — 800 рядків без можливості відкату | Коміт після кожного верифікованого кроку |
| Без `/think` | Правдоподібний але архітектурно невірний код | `think hard` для ризикових рішень |
| Scope creep | Переписані класи, яких не просили чіпати | "minimal change" у промпті + `reviewer` |
| Silent Failures | Новий `try-catch` у diff | "Re-throw, do not swallow" у промпті |
| Нескінченний retry | 5+ спроб без прогресу | `/plan` + переформулювання задачі |
