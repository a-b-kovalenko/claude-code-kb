## 📝 TL;DR

Питання та відповіді для самоперевірки по Domain 1: Agentic Architecture (27%). Формат іспиту — сценарій + 4 варіанти, один правильний. Пояснення "чому" важливіше за саму відповідь.

## Питання 1

Оркестратор запускає субагента-дослідника і субагента-реалізатора. Дослідник знаходить, що цільовий API підтримує лише POST запити, але оркестратор не передає цю інформацію реалізатору. Реалізатор генерує GET запити і отримує помилки. Яка root cause?

**A)** Реалізатор не має доступу до документації API  
**B)** Субагент context not passed explicitly від оркестратора  
**C)** API не підтримує агентські виклики  
**D)** Оркестратор має використовувати flat multi-agent замість hub-and-spoke

<details>
<summary>Правильна відповідь</summary>

**B** — Типовий failure pattern домену: оркестратор "знає" факт (тільки POST), але не передав його явно субагенту. Реалізатор почав з нуля і вгадував. Виправлення: кожен виклик субагента містить повний необхідний контекст у своєму input.

</details>

## Питання 2

Агент виконує міграцію даних. На 847-му записі з 1200 він падає через мережеву помилку. Після перезапуску агент починає з запису 1. Яка архітектурна проблема?

**A)** Відсутній retry механізм для мережевих помилок  
**B)** Агент не зберігає checkpoint після кожного обробленого запису  
**C)** Потрібен окремий субагент для error handling  
**D)** Міграцію треба виконувати в транзакції

<details>
<summary>Правильна відповідь</summary>

**B** — Відсутність session resumption. Checkpoint після кожного кроку (або батчу) дозволяє продовжити з місця зупинки, а не спочатку. A — частково вірно, але не вирішує проблему повторного початку.

</details>

## Питання 3

Команда будує систему з 5 субагентів. Один розробник пропонує дозволити агентам спілкуватись напряму між собою (peer-to-peer) для зменшення latency. Який головний ризик?

**A)** Збільшення вартості API викликів  
**B)** Складність дебагінгу та відсутність єдиного контролю стану  
**C)** API Claude не підтримує peer-to-peer комунікацію  
**D)** Latency не зменшиться — тільки збільшиться

<details>
<summary>Правильна відповідь</summary>

**B** — Flat multi-agent ускладнює: дебагінг (хто кому що сказав?), відновлення після помилок, контроль стану. Hub-and-spoke — рекомендований патерн за замовчуванням. A і D — не є головними ризиками.

</details>

## Питання 4

Агент має виконати незворотну операцію: видалення 10 000 записів з продакшн бази. Як правильно сконфігурувати поведінку?

**A)** Запустити агента з `--no-interactive` і дозволити самостійно виконати  
**B)** Додати hook що блокує будь-які DELETE операції  
**C)** Визначити escalation trigger для незворотних дій — зупинитись і запитати підтвердження  
**D)** Використати plan mode щоб агент спочатку склав список записів

<details>
<summary>Правильна відповідь</summary>

**C** — Незворотні дії — один з трьох валідних escalation triggers. Агент повинен зупинитись і передати контроль людині. B — занадто жорстко (блокує всі DELETE). D — не вирішує проблему підтвердження.

</details>

## Питання 5

Оркестратор запускає агента-дослідника і чекає результату. Дослідник не повертає відповідь протягом 10 хвилин (зависає в нескінченному циклі). Яке правильне рішення?

**A)** Збільшити timeout оркестратора до 30 хвилин  
**B)** Визначити max_iterations для субагента + escalation при перевищенні  
**C)** Перейти на flat multi-agent архітектуру  
**D)** Додати більше пам'яті до контекстного вікна субагента

<details>
<summary>Правильна відповідь</summary>

**B** — Неконтрольований retry/iteration loop — типовий failure pattern. Виправлення: явний ліміт ітерацій + escalation trigger. A — відкладає проблему, не вирішує. D — не стосується.

</details>
