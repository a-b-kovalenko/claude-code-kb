## 📝 TL;DR

Найбільший домен (27%): архітектура агентських систем у production. Основні теми — оркестрація, декомпозиція задач, session resumption, hooks vs. prompts. Більшість помилок — це втрачений контекст між субагентами.

## Патерни оркестрації

### Hub-and-spoke

Центральний оркестратор керує субагентами і зберігає стан:

```text
Orchestrator
├── Agent A (дослідження)
├── Agent B (реалізація)
└── Agent C (верифікація)
```

**Переваги:** простий контроль, єдиний стан, легко retry.  
**Ризик:** оркестратор стає вузьким місцем.

### Flat multi-agent (peer-to-peer)

Агенти спілкуються між собою напряму. Складніше контролювати, важче дебажити. **Рідко виправдано** в production без чіткого протоколу.

**Правило:** за замовчуванням — hub-and-spoke. Flat — тільки якщо є чіткий message-passing протокол.

## Декомпозиція задач

Хороша декомпозиція:

- Кожна підзадача має **чіткий вхід і вихід**
- Субагент отримує **лише необхідний контекст** (не весь стан оркестратора)
- Результат субагента — структурований (JSON, не free-text)

Погана декомпозиція: "зроби все що потрібно для фічі X" — агент не знає де зупинитись.

## Session Resumption

Якщо агент падає посередині — як продовжити?

- Зберігати **checkpoint** після кожного завершеного кроку
- Checkpoint містить: поточний крок, результати попередніх кроків, змінний стан
- При старті — перевіряти checkpoint і пропускати виконані кроки

**Типова помилка:** зберігати тільки фінальний результат — при падінні все починається з нуля.

## Hooks vs. Prompts

| | Hooks | Prompts |
| --- | --- | --- |
| Коли спрацьовує | На події (pre/post tool use) | За інструкцією в тексті |
| Гарантія виконання | Висока (shell script) | Середня (LLM може проігнорувати) |
| Типовий use case | Валідація, захист, сайд-ефекти | Бізнес-логіка, умовна поведінка |
| Де конфігурується | `settings.json` | `CLAUDE.md`, system prompt |

**Правило:** якщо потрібна **гарантія** виконання — hook. Якщо потрібна **гнучкість** — prompt.

## Self-Review Limitation

Модель що переглядає власний вивід **у тій самій сесії** зберігає ланцюг свого попереднього міркування. Це систематична упередженість — модель підтверджує, а не оскаржує свої рішення.

```text
ПОГАНО — self-review в тій самій сесії:
  instance → output → same instance reviews → confirms original reasoning

ДОБРЕ — незалежний інстанс:
  instance A → output → instance B (без prior context) → fresh evaluation
```

Незалежний інстанс підходить до виводу без контексту "чому я так вирішив" — суттєво вищий рівень виявлення проблем.

**Типова пастка на іспиті:** "run three passes on the full PR and vote by majority" — три проходи з тими самими системними пропусками в увазі матимуть однакові сліпі плями.

## Production Architecture: 5-Step Review

```text
1. Generation instance    → створює вивід
2. Per-unit review        → кожен файл/документ окремим інстансом
3. Integration instance   → cross-unit: data flow, суперечності, API contracts
4. Confidence routing     → findings → direct_report | human_review
5. Calibration loop       → коригує пороги за результатами labeled validation
```

Більше контекстне вікно не вирішує проблему якості уваги. Модель, що аналізує 14 файлів у великому вікні, все одно розпорошує увагу — задача залишається надто широкою для одного проходу.

## Production Failure Patterns

### 1. Subagent context not passed explicitly

Оркестратор "знає" контекст, але не передає його субагенту явно. Субагент починає здогадуватися — і помиляється.

**Виправлення:** кожен виклик субагента містить повний необхідний контекст у своєму prompt/input.

### 2. Неконтрольований retry loop

Агент отримує помилку → повторює → знову помилка → нескінченний цикл.

**Виправлення:** максимальна кількість спроб + escalation trigger при перевищенні.

### 3. Втрата стану між сесіями

Агент завершує сесію, наступна починається з нуля без знання попередніх результатів.

**Виправлення:** явне збереження стану у файлі або structured output перед завершенням сесії.
