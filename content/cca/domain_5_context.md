## 📝 TL;DR

Найменший домен (15%), але помилки тут каскадують на всі інші. Три теми: progressive summarisation ризики, escalation triggers, provenance tracking у multi-agent системах.

## Progressive Summarisation — ризики

Claude Code автоматично стискає контекст коли він переповнюється (`/compact`). Проблема: деякі факти не виживають стискання.

**Що втрачається:**

- Точні числові значення (ID, версії, дати)
- Деталі помилок і stack traces
- Проміжні рішення ("ми спробували X але відмовились бо...")
- Явні інструкції із середини розмови

**Типова помилка:** транзакційні факти (номер рахунку, order ID) зберігаються у вільному тексті розмови → стискаються → агент продовжує з неправильними даними.

**Виправлення:** критичні факти закріплювати у структурованих артефактах (файлах, JSON), не покладатися на пам'ять розмови.

## Escalation Triggers

Три валідні причини для escalation (передача контролю людині або оркестратору):

1. **Досягнуто ліміт спроб** — агент не зміг виконати задачу після N повторів
2. **Виявлено невизначеність у вимогах** — агент не може вирішити між двома рівнозначними підходами
3. **Ризик незворотної дії** — операція, яку неможливо відкотити (delete, deploy, payment)

**Правило:** агент не повинен продовжувати "в надії що вийде". Краще зупинитись і ескалувати.

**Типова помилка:** агент продовжує після помилки без ескалації — накопичує пошкодження.

## Provenance Tracking

У multi-agent системах кожен результат повинен містити метадані про своє походження:

```json
{
  "result": "...",
  "provenance": {
    "agent": "researcher-agent-1",
    "source": "docs.anthropic.com/agents",
    "timestamp": "2026-06-21T10:00:00Z",
    "confidence": "high"
  }
}
```

**Навіщо:** якщо фінальний результат оркестратора виявився неправильним — можна відстежити який субагент надав невірні дані.

**Типова помилка:** субагент повертає тільки результат без джерела — при помилці неможливо визначити причину.

## Structured Error Propagation

Помилки мають передаватись структуровано, а не як рядок тексту:

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "field": "email",
    "message": "Invalid format",
    "retryable": true
  }
}
```

Це дозволяє оркестратору:

- Вирішити чи робити retry
- Передати точну інформацію про помилку далі по ланцюгу
- Логувати структуровані events

## Production Failure Patterns

### 1. Progressive summarisation destroying transactional facts

Агент обробляв великий список замовлень. Після стискання контексту — забув які вже оброблено.

**Виправлення:** checkpoint файл з ID оброблених записів, оновлюється після кожного батчу.

### 2. Escalation не визначена — агент "застрягає"

Агент повторює одну і ту ж дію 10 разів у надії на інший результат.

**Виправлення:** явний `max_retries` + escalation handler у оркестраторі.

### 3. Непрозорий error propagation

Субагент повертає `{"error": "something went wrong"}` — оркестратор не знає чи це retryable.

**Виправлення:** завжди включати `retryable: boolean` та `code` у структуру помилки.
