## Питання 1

Команда з 5 розробників використовує Claude Code. Тімлід налаштував детальні coding conventions у `~/.claude/CLAUDE.md` на своїй машині. Нові члени команди скаржаться що агент ігнорує стандарти. В чому проблема?

**A)** Файл `~/.claude/CLAUDE.md` не підтримується Claude Code  
**B)** Командні стандарти треба розміщувати у `<project>/CLAUDE.md`, а не в user-level config  
**C)** Потрібно додати стандарти до system prompt через API  
**D)** Кожен розробник має скопіювати файл вручну

<details>
<summary>Правильна відповідь</summary>

**B** — `~/.claude/CLAUDE.md` — user-level, видимий лише поточному користувачу. Командні стандарти мають бути у `<project>/CLAUDE.md` (в репо) — тоді всі члени команди їх бачать автоматично.

</details>

## Питання 2

Проєкт має специфічні правила для тестових файлів (заборона моків, обов'язковий Testcontainers). Де найкраще розмістити ці правила щоб вони застосовувались лише при роботі з тестами?

**A)** У `<project>/CLAUDE.md` в секції "Testing"  
**B)** У `.claude/rules/tests.md` з glob `**/*.test.java`  
**C)** У `~/.claude/CLAUDE.md` кожного розробника  
**D)** У коментарях до кожного тестового файлу

<details>
<summary>Правильна відповідь</summary>

**B** — Path-specific rules через glob завантажуються лише коли агент працює з відповідними файлами. Це зменшує шум у контексті. A — правило завжди завантажується, навіть для не-тестових файлів.

</details>

## Питання 3

CI/CD pipeline запускає Claude Code командою `claude "виправ lint помилки"`. Pipeline зависає і не завершується. Яка найімовірніша причина?

**A)** Claude Code не підтримується у CI середовищі  
**B)** Відсутній флаг `--print` або `--no-interactive` — агент чекає на інтерактивний ввід  
**C)** Lint помилок немає, тому агент не знає що робити  
**D)** Потрібна аутентифікація через OAuth замість API key

<details>
<summary>Правильна відповідь</summary>

**B** — Без `--print` Claude Code запускає інтерактивний REPL і чекає на людину. У CI завжди потрібен `--print` або `--no-interactive`. A — хибне, CI підтримується. D — не стосується.

</details>

## Питання 4

Розробник хоче що агент використовував plan mode для всіх змін у `src/core/` але direct execution для `src/utils/`. Як це налаштувати?

**A)** Запускати агента з різними флагами вручну для кожної директорії  
**B)** Створити два файли у `.claude/rules/` з різними glob патернами і відповідними інструкціями  
**C)** Це неможливо — plan mode або увімкнений глобально або ні  
**D)** Додати умову в `CLAUDE.md`: "якщо файл у src/core — використовуй plan mode"

<details>
<summary>Правильна відповідь</summary>

**B** — `.claude/rules/core-rules.md` з glob `src/core/**` і інструкцією завжди використовувати `/plan`. `.claude/rules/utils-rules.md` з glob `src/utils/**` без обмежень. D — може спрацювати, але B надійніший (правило завантажується тільки при потребі).

</details>

## Питання 5

Агент отримав завдання "оновити всі залежності в проєкті". Він оновив 47 файлів і зламав 3 критичні модулі. Яке налаштування запобігло б цьому?

**A)** Зменшити розмір контекстного вікна  
**B)** Використати plan mode — агент спочатку склав би план і показав scope змін  
**C)** Заборонити агенту писати у файли через settings.json  
**D)** Запустити агента з `--dry-run` флагом

<details>
<summary>Правильна відповідь</summary>

**B** — Plan mode → агент досліджує, пише план, зупиняється і чекає підтвердження. Людина бачить scope до виконання. C — надто жорстко. D — `--dry-run` не є стандартним флагом Claude Code.

</details>
