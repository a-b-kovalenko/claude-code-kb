## 📝 TL;DR

Критерій вибору між Batch API і real-time — не схема даних, а latency-вимоги. Однакова JSON-схема не означає однаковий pipeline: термінові документи з SLA 30 хвилин йдуть через real-time API, решта — через Batch API зі знижкою 50%.

## Питання

Система обробляє два типи документів з однаковою JSON-схемою:

- **Стандартні місячні звіти** — архівуються після обробки, без вимог до latency
- **Термінові exception reports** — мають запускати бізнес-алерти протягом 30 хвилин з моменту отримання

Як архітектурно побудувати pipeline обробки?

**A)** Чергувати всі документи і відправляти щогодинними батчами, позначаючи термінові для прискореної обробки коли повернуться результати батчу.

**B)** Відправляти всі документи до Batch API з `custom_id` для відстеження.

**C)** Маршрутизувати стандартні звіти до Batch API для 50% економії, а термінові exception reports — до real-time Messages API.

**D)** Відправляти всі документи до real-time Messages API для однакової latency між типами документів.

## Правильна відповідь: C

## Аналіз варіантів

### A — Хибний

Щогодинні батчі означають, що терміновий документ може чекати до 60 хвилин лише до відправки батчу — SLA в 30 хвилин порушується ще до початку обробки. "Флаг для прискорення" не змінює механіку Batch API.

### B — Хибний

Batch API обробляє запити асинхронно до 24 годин. `custom_id` — механізм зіставлення відповідей, не пріоритизації. Термінові документи не можуть чекати годинами.

### C — Правильний

Маршрутизація за latency-вимогами, а не за схемою:

- Стандартні звіти: latency не критична → Batch API, 50% знижка
- Термінові reports: SLA 30 хвилин → real-time API, відповідь за секунди

Однакова JSON-схема спрощує реалізацію — той самий парсер обробляє обидва потоки.

### D — Хибний

Real-time для всіх документів вдвічі дорожче без жодної переваги для стандартних звітів. "Однакова latency" — не бізнес-вимога; вимога лише для термінових документів.

## Ключові концепції

### Критерій вибору Batch API vs real-time

| Характеристика | Batch API | Real-time API |
| --- | --- | --- |
| Вартість | 50% знижка | Повна ціна |
| Час обробки | До 24 годин | Секунди |
| Підходить для | Архівування, аналітика, датасети | SLA < годин, алерти, інтерактив |

### Hybrid routing pattern

Один pipeline, два шляхи на основі атрибутів документа:

```text
document received
  → classify: urgent? → real-time API → alert trigger
  → classify: standard? → Batch API → archive
```

Однакова схема відповіді — той самий downstream парсер для обох шляхів.

### Ключовий принцип

Схема даних і transport — незалежні рішення. Однакова JSON-схема не зобов'язує використовувати однаковий API. Вибір транспорту визначається виключно latency-вимогами та бюджетом.
