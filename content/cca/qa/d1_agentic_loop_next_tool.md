## 📝 TL;DR

Agentic loop не має вбудованої маршрутизації: результат кожного інструменту додається до контексту розмови, і модель сама вирішує наступний крок. Немає ні фіксованої послідовності, ні decision tree — лише динамічне міркування на основі накопиченого контексту.

## Питання

Агент працює в agentic loop підтримки клієнтів. Він викликає `lookup_order` і отримує деталі замовлення: товар куплено 45 днів тому. Як agentic loop вирішує — викликати `process_refund` чи `escalate_to_human` далі?

**A)** Оркестраційний шар автоматично маршрутизує до наступного інструменту на основі поля `status` замовлення.

**B)** Агент виконує кроки послідовності інструментів, запланованої на початку запиту.

**C)** Агент слідує попередньо налаштованому decision tree, що зіставляє атрибути замовлення з конкретними викликами інструментів.

**D)** Деталі замовлення додаються до розмови, і модель міркує про наступну дію.

## Правильна відповідь: D

## Аналіз варіантів

### A — Хибний

Немає "оркестраційного шару" що автоматично читає поля й маршрутизує. Маршрутизація — це і є завдання моделі, не окремого компонента.

### B — Хибний

Agentic loop не фіксує план інструментів на старті. Послідовність формується динамічно — кожен наступний крок залежить від результату попереднього.

### C — Хибний

Decision tree — це hardcoded логіка поза моделлю. У agentic loop рішення приймає сама модель, а не зовнішнє правило.

### D — Правильний

Результат `lookup_order` (деталі замовлення, дата покупки) додається як assistant/tool message до розмови. Модель бачить повний контекст — оригінальний запит, системний промпт з правилами повернень, результат інструменту — і вирішує: 45 днів > порогу повернення → `escalate_to_human`.

## Ключові концепції

### Механізм agentic loop

```text
user message
  → model → tool call (lookup_order)
  → tool result додається до контексту
  → model бачить оновлений контекст → наступний tool call або відповідь
```

Кожна ітерація: модель отримує повний контекст (включно з усіма попередніми результатами інструментів) і вирішує що далі. Це і є "reasoning" — не маршрутизація.

### Чому це важливо для дизайну

- Системний промпт визначає **правила** (напр. "повернення лише до 30 днів")
- Модель **застосовує** ці правила до конкретних даних з інструменту
- Розробник не кодує if/else — він формулює правила в промпті
