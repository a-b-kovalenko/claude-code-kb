## 📝 TL;DR

Непослідовна екстракція конкретного поля — сигнал що модель не має чіткого розуміння формату, а не що їй бракує потужності. Few-shot приклади з повними парами input-output встановлюють неявні правила екстракції, які узагальнюються на різні стилі опису товарів.

## Питання

Система парсить описи e-commerce продуктів і витягує специфікації (dimensions, weight, materials) у JSON. Поле `materials` витягується непослідовно.

Який найефективніший спосіб покращити послідовність екстракції?

**A)** Зробити поле `materials` обов'язковим у схемі замість опціонального — щоб модель завжди виводила значення.

**B)** Перейти на потужнішу модель, оскільки непослідовна екстракція свідчить про недостатні можливості моделі.

**C)** Встановити temperature=0 щоб усунути рандомність і забезпечити детермінований вивід.

**D)** Додати few-shot приклади з 2-3 повними парами input-output зі стандартизованим форматом опису матеріалів.

## Правильна відповідь: D

## Аналіз варіантів

### A — Хибний

`required` у схемі змушує модель **завжди вивести** значення поля, але не вчить **як** його витягувати. Якщо опис товару неоднозначний, модель може галюцинувати або витягнути невірний формат. Проблема — не відсутність значення, а неоднозначність інтерпретації.

### B — Хибний

Непослідовність не означає брак потужності. Модель іноді витягує `materials` правильно — значить, здатність є. Причина — варіативність форматів вхідних описів, а не ліміт моделі. Перехід на сильнішу модель дорого і не вирішить проблему формату.

### C — Хибний

Temperature=0 усуває рандомність генерації, але не вирішує проблему неоднозначності вхідних даних. Якщо модель не знає як інтерпретувати "materials" у різних контекстах, вона детерміновано робитиме одну й ту саму помилку для певних inputs.

### D — Правильний

Few-shot приклади показують модель точно що вважається "правильним" форматом `materials` у різних ситуаціях. 2-3 пари input-output охоплюють варіативність (тканини, метали, пластики, composites) і встановлюють неявний стандарт що узагальнюється на нові описи.

## Ключові концепції

### Коли застосовувати few-shot для екстракції

Few-shot найефективніший коли:

- Поле витягується непослідовно (не ніколи, а іноді)
- Вхідні дані мають варіативний формат
- Є чіткий стандарт "правильного" виводу

### Структура few-shot для екстракції

```text
Приклад 1:
Input: "Made from 100% organic cotton with polyester lining"
Output: { "materials": ["cotton", "polyester"] }

Приклад 2:
Input: "Stainless steel frame, tempered glass surface"
Output: { "materials": ["stainless steel", "tempered glass"] }

Приклад 3:
Input: "Solid oak wood, brass hardware"
Output: { "materials": ["oak", "brass"] }
```

2-3 прикладів достатньо щоб модель зрозуміла: масив рядків, нижній регістр, без одиниць вимірювання.

### Діагностика причини непослідовності

| Симптом | Причина | Рішення |
| --- | --- | --- |
| Поле ніколи не витягується | Модель не знає що шукати | Опис поля в схемі / промпті |
| Поле витягується іноді | Неоднозначний формат входу | Few-shot приклади |
| Поле завжди неправильного типу | Схема не відповідає даним | Уточнити тип у schema |
| Поле завжди порожнє | Дані відсутні у вхідних текстах | Перевірити вхідні дані |
