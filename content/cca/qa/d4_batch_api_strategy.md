## 📝 TL;DR

При витязі структурованих даних з 50 000 юридичних контрактів Batch API дає 50% знижку, але лише якщо промпти відпрацьовані заздалегідь. Оптимальна стратегія: валідуй на ~1 000 зразках через real-time API, виправ промпти — і лише тоді запускай повний batch. Два тижні дедлайну дозволяють кілька таких циклів.

## Питання

Твоя команда витягує структуровані дані з 50 000 legacy-юридичних контрактів. Дедлайн — два тижні. Batch API: 50% знижка, вікно обробки до 24 годин. Яка стратегія найбільш cost-efficient при дотриманні дедлайну?

**A)** Відправити всі документи через Batch API, невалідні — повторно у наступних батчах з уточненням промптів, поки всі не пройдуть валідацію.

**B)** Використати real-time API для всіх 50 000 документів (24-годинне вікно Batch API недостатнє).

**C)** Обробити 1 000 зразків через real-time API, виявити патерни відмов і уточнити промпти — потім відправити всі 50 000 через Batch API з оптимізованими промптами.

**D)** Розбити на послідовні батчі, порівнювати результати і поступово покращувати якість між ними.

## Правильна відповідь: C

## Аналіз варіантів

### A — Хибний

Якщо промпти не відпрацьовані, перший batch поверне масово невалідні результати. Повторні batch-запуски з корекцією — витрата часу і квоти: кожен цикл займає до 24 годин.

### B — Хибний

Real-time API для 50k документів вдвічі дорожче ніж Batch API. Якщо дедлайн дозволяє 24h вікно — вибір real-time нічим не виправданий.

### C — Правильний

1 000 зразків через real-time → швидко (хвилини) і дешево → виявляємо проблеми промптів → виправляємо → батчимо 50k одним запуском. Дедлайн виконується, витрати мінімальні.

### D — Хибний

Послідовні батчі без чіткої зупинки — варіант A в іншому формулюванні. Поступове покращення між батчами може тривати кілька днів і не гарантує завершення вчасно.

## Ключові концепції

### Anthropic Message Batches API

Окремий асинхронний endpoint Anthropic для масової обробки незалежних запитів. Замість того щоб надсилати 50 000 запитів по одному і чекати на кожну відповідь, ти пакуєш їх усі в один batch-запит — API обробляє їх у фоні й повертає результати одним файлом.

Типові use cases: витяг структурованих даних з юридичних або фінансових документів, класифікація великих архівів, масова генерація резюме, анотування датасетів — будь-що, де відповідь потрібна не "зараз", а "до дедлайну".

| Параметр | Значення |
| --- | --- |
| Знижка | 50% від ціни real-time API |
| Вікно обробки | До 24 годин (асинхронно) |
| Use case | Великі обсяги незалежних запитів без вимог до latency |
| Обмеження | Не підходить для залежних задач або там де відповідь потрібна негайно |

### Стратегія "sample-first, batch-later"

1. Відбери ~1–2% від загального обсягу як зразок
2. Запусти через real-time API — отримай відповідь негайно
3. Знайди патерни помилок, уточни промпти
4. Лише після валідації — запускай повний batch

Batch-запуск без попередньої валідації промптів означає оплату за масово невалідні результати.

## Технічна реалізація

### Lifecycle батчу

```text
POST /v1/messages/batches                → отримуєш batch_id
GET  /v1/messages/batches/{id}           → polling: processing_status = "in_progress" | "ended"
GET  /v1/messages/batches/{id}/results   → JSONL-файл з результатами
```

Кожен запит у батчі має `custom_id` (твій ідентифікатор) — по ньому зіставляєш відповідь з оригінальним запитом.

### Ліміти

| Параметр | Значення |
| --- | --- |
| Запитів на батч | До 10 000 або 32 MB (менше з двох) |
| Термін зберігання результатів | 29 днів |
| Скасування | Можливе до завершення обробки (`DELETE /v1/messages/batches/{id}`) |

### Що не підходить для Batch API

- Залежні запити (результат одного потрібен для наступного)
- Streaming
- Інтерактивні сценарії з вимогою до latency
- Tool use з негайним виконанням інструментів
