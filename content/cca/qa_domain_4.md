## Питання 1

Система має два інструменти: `search_docs` ("searches content") і `search_code` ("searches content"). Агент постійно викликає `search_docs` замість `search_code` при роботі з Java файлами. Яка причина?

**A)** Агент не має доступу до файлової системи  
**B)** Обидва описи однаково розпливчасті — модель не може розрізнити інструменти  
**C)** Потрібно перейменувати інструменти на більш очевидні назви  
**D)** Проблема у порядку реєстрації інструментів

<details>
<summary>Правильна відповідь</summary>

**B** — Vague tool descriptions causing misrouting — типовий failure pattern. Опис має явно вказувати тип даних: "searches only `.java`, `.ts` source files" vs "searches Markdown and AsciiDoc documentation". C допомагає, але недостатньо без хорошого опису.

</details>

## Питання 2

Команда будує MCP сервер з API ключем до зовнішнього сервісу. Де правильно зберігати конфігурацію з ключем?

**A)** У `.mcp.json` в кореневій директорії проєкту (project scope)  
**B)** У `.claude/mcp.json` (local scope, не в репо) або через env variable  
**C)** У `~/.claude/mcp.json` кожного розробника  
**D)** Закодувати ключ у вихідному коді сервера

<details>
<summary>Правильна відповідь</summary>

**B** — Local scope (`.claude/mcp.json`) не потрапляє в репо. Альтернатива — env variable з посиланням `${API_KEY}` у конфіг файлі. A — `.mcp.json` зазвичай в репо → секрет витікає. D — критична помилка безпеки.

</details>

## Питання 3

Агент має обробити 500 незалежних документів і для кожного викликати інструмент `analyze_document`. Який підхід оптимальний?

**A)** Викликати інструменти послідовно — один за одним  
**B)** Batch execution — викликати інструменти паралельно для незалежних операцій  
**C)** Розбити на 5 субагентів по 100 документів кожен  
**D)** Використати streaming для зменшення latency

<details>
<summary>Правильна відповідь</summary>

**B** — Незалежні операції → batch (паралельне виконання). Зменшує загальний час у N разів. C може бути вірним для дуже великих обсягів, але B — правильна відповідь для базового сценарію іспиту. A — неоптимально. D — не стосується tool execution.

</details>

## Питання 4

MCP сервер надає доступ до бази даних для читання. Яким примітивом MCP найкраще це реалізувати?

**A)** Tool — `get_records(table, filters)`  
**B)** Resource — `db://table_name`  
**C)** Prompt — шаблон запиту до бази  
**D)** Tool або Resource — різниці немає

<details>
<summary>Правильна відповідь</summary>

**B** — Resources — для read-only доступу до даних (файли, БД, API). Tools — для дій зі сайд-ефектами. Якщо операція тільки читає дані і не змінює стан — Resource є семантично правильним вибором. D — різниця є і важлива для іспиту.

</details>

## Питання 5

Tool опис: "Processes the input and returns output." Що конкретно потрібно додати щоб модель правильно використовувала інструмент?

**A)** Збільшити довжину опису вдвічі  
**B)** Що саме робить інструмент, коли його використовувати і що повертає  
**C)** Додати параметр `usage_examples` до schema  
**D)** Перейменувати інструмент у більш описову назву

<details>
<summary>Правильна відповідь</summary>

**B** — Хороший опис містить три елементи: що робить (конкретна дія), коли використовувати (use case, відмінності від схожих інструментів), що повертає (формат output). D допомагає, але назва без хорошого опису — недостатньо.

</details>
