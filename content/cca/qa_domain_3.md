Питання та відповіді для самоперевірки по Domain 3: Prompt Engineering & Structured Output (20%). Підступний домен — неправильні відповіді звучать переконливо. Фокус на JSON schema та retry logic.

## Питання 1

Система класифікує підтримуючі тікети і зберігає результат у полі `priority`. Іноді модель повертає "urgent", "critical", "high-priority" замість очікуваних "low", "medium", "high". Яке виправлення?

**A)** Додати у промпт: "відповідай лише low, medium або high"  
**B)** Визначити `enum: ["low", "medium", "high"]` у JSON schema для поля `priority`  
**C)** Фільтрувати результати post-processing скриптом  
**D)** Використати few-shot з прикладами правильних значень

<details>
<summary>Правильна відповідь</summary>

**B** — `enum` у JSON schema — найнадійніший спосіб обмежити значення поля. Модель не може повернути значення поза enum без порушення схеми (валідація відловить). A і D — зменшують проблему, але не гарантують. C — хак, не рішення.

</details>

## Питання 2

Validation-retry loop повертає помилку 3 рази підряд з однаковою відповіддю. Що найімовірніше не так?

**A)** Модель не підтримує JSON output  
**B)** Retry prompt не містить опису що саме не так і чого очікується  
**C)** Потрібно збільшити `max_tokens` у параметрах  
**D)** JSON schema занадто складна для моделі

<details>
<summary>Правильна відповідь</summary>

**B** — Retry без пояснення помилки = модель повторює те саме. Правильний retry включає: що повернула модель, чому це не відповідає схемі, що очікується. A і D — малоймовірно для Claude. C — не стосується валідації.

</details>

## Питання 3

Промпт для вилучення дат з документів: "Знайди дати в тексті і поверни їх. Відповідай тільки якщо впевнений." Система повертає вигадані дати. Яка причина?

**A)** Модель галюцинує через недостатньо даних у документі  
**B)** "Відповідай тільки якщо впевнений" — суб'єктивна інструкція, модель завжди вважає себе впевненою  
**C)** Потрібно використати більшу модель  
**D)** Промпт треба написати англійською

<details>
<summary>Правильна відповідь</summary>

**B** — Vague confidence instruction — типовий failure pattern. Виправлення: конкретний критерій — "якщо в тексті немає явної дати у форматі DD.MM.YYYY або Month DD, YYYY — поверни `null`".

</details>

## Питання 4

Система витягує дані про замовлення. Поле `discount_code` необов'язкове. Розробник очікує `null` коли знижки немає, але іноді поле відсутнє взагалі. JSON schema визначена як: `"discount_code": { "type": "string" }` без `required`. Яка проблема?

**A)** Поле потрібно додати до `required` масиву  
**B)** Необов'язкове поле може бути відсутнє (undefined), а не `null` — треба визначити `"type": ["string", "null"]`  
**C)** Це правильна поведінка, код має перевіряти і `null` і відсутність  
**D)** Потрібно використати `default: null` у schema

<details>
<summary>Правильна відповідь</summary>

**B** — Якщо поле не в `required`, воно може бути відсутнє зовсім. Якщо потрібен явний `null`, треба `"type": ["string", "null"]` і додати до `required`. D — `default` у JSON schema не примушує модель повертати значення. C — технічно вірно, але не відповідає на питання про причину.

</details>

## Питання 5

Потрібно навчити модель класифікувати відгуки на три категорії. Few-shot містить 4 приклади — всі позитивні відгуки з правильною класифікацією. Модель погано класифікує негативні відгуки. Що додати?

**A)** Збільшити кількість позитивних прикладів до 10  
**B)** Додати приклади з негативними відгуками та edge cases  
**C)** Перейти на chain-of-thought промптинг  
**D)** Описати категорії детальніше у системному промпті

<details>
<summary>Правильна відповідь</summary>

**B** — Few-shot без негативних прикладів — модель не бачить як обробляти edge cases. Збалансовані приклади (позитивні + негативні + межові) дають значно кращий результат. D — може допомогти, але B вирішує проблему напряму.

</details>
