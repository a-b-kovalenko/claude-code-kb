## 📝 TL;DR

Headless-режим — запуск Claude Code без інтерактивного REPL через прапор `--print`: агент отримує один промпт, виконує задачу і завершується з результатом у stdout. Це перетворює Claude Code на компонент pipeline — як `grep` або `jq`. Для кожної незалежної задачі — окремий процес; `--max-turns` обов'язковий щоб уникнути нескінченних циклів.

## 🤔 Що таке headless-режим

У звичайному режимі відбувається діалог: ти пишеш, Claude відповідає, ти відповідаєш знову. У headless-режимі Claude отримує один промпт, виконує задачу і завершується — як будь-яка CLI-утиліта (`grep`, `jq`, `curl`).

| | Звичайний | Headless |
| --- | --- | --- |
| Запуск | `claude` | `claude --print "..."` |
| Взаємодія | Діалог | Один запит → відповідь |
| Вивід | TUI у терміналі | stdout (текст або JSON) |
| Використання | Локальна розробка | CI, скрипти, автоматизація |

Вивід можна передати в `jq`, записати у файл або паралельно запустити кілька інстанцій.

## ✅ Які задачі підходять для headless

Headless добре вирішує задачі з **чітким входом і виходом** — коли не потрібен діалог для уточнень.

| Задача | Приклад |
| --- | --- |
| Автоматичний PR review | Баги, SOLID, security — коментар до кожного PR |
| Security scan | OWASP Top 10 при push у main |
| Валідація міграцій | Перевірити що новий Flyway SQL не руйнує схему |
| Генерація changelog | З `git diff` між тегами |
| Перевірка commit-повідомлень | Відповідність Conventional Commits |
| Javadoc до публічного API | Автоматично при зміні інтерфейсів |

**Загальний критерій:** задача повторювана, добре визначена і не потребує рішення людини в процесі.

**Що погано лягає в headless:**

- Складний рефакторинг — агент може зайти в глухий кут без можливості запитати
- Налагодження незрозумілої помилки — потрібна ітерація
- Будь-що де потрібне рішення людини посередині

## ⚡ Одна задача — один процес

Не варто об'єднувати кілька незалежних задач в один промпт ("зроби review, перевір security і перевір commit message"). Краще — окремий процес на кожну, запущені паралельно:

```bash
claude --print "Review the diff for bugs and SOLID violations..." \
  --model claude-sonnet-4-6 > /tmp/review.md &

claude --print "Scan src/ for OWASP Top 10 vulnerabilities..." \
  --model claude-haiku-4-5-20251001 > /tmp/security.md &

claude --print "Check if the commit message follows Conventional Commits..." \
  --model claude-haiku-4-5-20251001 > /tmp/commit.md &

wait
```

Переваги:

- **Паралельність** — три задачі виконуються одночасно, а не послідовно
- **Ізоляція** — падіння одного не блокує інші
- **Фокус** — вузький промпт дає точнішу відповідь
- **Різні моделі** — Sonnet для складного review, Haiku для простих перевірок

Принцип той самий що і в Unix pipes: одна відповідальність на процес.

## ⚙️ Ключові прапори

| Прапор | Призначення |
| --- | --- |
| `--print` / `-p` | Single-shot відповідь без REPL; вивід у stdout |
| `--output-format json` | JSON з полями `result`, `cost_usd`, `session_id` |
| `--output-format stream-json` | Потоковий JSON — по одному об'єкту на рядок |
| `--max-turns N` | Ліміт кроків агента; захист від нескінченного циклу |
| `--model <id>` | Явна модель (Haiku для дешевих задач, Sonnet для складних) |
| `--no-interactive` | Явна заборона REPL — корисно в скриптах |
| `--allowedTools` | Обмежити набір інструментів агента |

Базовий виклик:

```bash
claude --print "Review this PR for security issues" \
  --output-format json \
  --max-turns 5 \
  --model claude-haiku-4-5-20251001
```

Exit-коди: `0` = успіх, `1` = помилка агента, `2` = невалідний ввід.

### Чому --max-turns обов'язковий

У headless-режимі Claude діє як агент: читає файли, виконує команди, аналізує результат — і так кілька кроків. Без ліміту він може зациклитись:

- **Retry loop** — компіляція падає, агент виправляє, знову падає з іншою помилкою, знову виправляє...
- **Невизначена задача** — промпт без чіткого критерію Done: "покращ код" — агент вічно знаходить що покращити
- **Self-correction spiral** — вносить зміну, перевіряє, вирішує що зробив неправильно, відкочує, повторює

В інтерактивному режимі ти бачиш це і зупиняєш вручну. У CI процес крутиться у фоні і витрачає API credits поки не впреться в timeout runner-а. `--max-turns 3–5` — обов'язковий запобіжник, не перестрахування.
