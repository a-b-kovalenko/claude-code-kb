## 📝 TL;DR

Субагенти — це спеціалізовані AI-інструменти з власним вікном контексту. Вони дозволяють делегувати "брудну" роботу (читання логів, пошук файлів), залишаючи головного агента зосередженим на архітектурі та логіці задачі. Claude Code має три вбудованих агенти (Explore, Plan, general-purpose), що спрацьовують автоматично; кастомні агенти доповнюють їх проєктною специфікою.

## Навіщо потрібні субагенти?

### 1. Ізоляція контексту (Context Hygiene)

Головний агент має обмежену пам'ять. Якщо він прочитає 50 файлів проєкту та 2000 рядків логів тестів, він почне "глючити" або втрачати нитку розмови. Субагент виконує цю роботу у **власному тимчасовому вікні**, повертаючи головному агенту лише стислий підсумок.

### 2. Спеціалізація та економія

Ми можемо використовувати різні моделі для різних ролей:

- **Haiku (дешева/швидка):** Для масового читання коду та пошуку (explorer).
- **Sonnet (розумна/дорожча):** Для складного аналізу помилок (test-runner) та рев'ю коду (reviewer).

### 3. Обмеження прав

Субагенти можуть мати обмежений набір інструментів (наприклад, тільки Read-only), що запобігає випадковому псуванню коду під час простого дослідження.

## Вбудовані субагенти

Claude Code має три вбудованих агенти, які спрацьовують **автоматично без будь-якої конфігурації**:

- **Explore** — Haiku, read-only, швидкий пошук по кодовій базі (не завантажує CLAUDE.md).
- **Plan** — read-only, збір контексту під час plan mode (не завантажує CLAUDE.md).
- **general-purpose** — усі інструменти, складні багатокрокові задачі.

Кастомний агент з `name: explorer` **не конфліктує** з вбудованим `Explore` — це різні ідентифікатори. Claude обирає між ними за `description`: більш специфічний опис переважає. Детальніше: [🤖 Вбудовані агенти](Builtin_Agents.md).

## Канонічна трійка субагентів (The Loop Trio)

| Субагент        | Модель | Роль та обов'язки                                                                                                                                                      |
| :-------------- | :----- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Explorer**    | Haiku  | **Дослідник.** Шукає файли, малює схеми залежностей, аналізує структуру БД. Повертає список файлів, які треба змінити.                                                 |
| **Test-runner** | Sonnet | **Діагност.** Запускає тести, аналізує стектрейси, шукає причину падіння. Повертає чітку інструкцію: "Падає тут, бо NullPointerException у рядку X".                   |
| **Reviewer**    | Sonnet | **Суддя.** Аналізує фінальний diff. Перевіряє відповідність стандартам `CLAUDE.md`, архітектурним межам та наявність тестів. Виносить вердикт: "Approve" або "Reject". |

## Де вони живуть?

- **Проєктні:** `.claude/agents/*.md` — спільні для всієї команди, комітяться в Git.
- **Персональні:** `~/.claude/agents/*.md` — ваші приватні помічники на всіх проєктах.

## Використання локальних моделей (Гібридний підхід)

Локальні моделі (через Ollama/LM Studio) — це чудові **виконавці рутини**, а не "мізки" системи.

- **Що віддавати локалці:** Генерацію Javadoc, нормалізацію імен тестів, форматування JSON, написання boilerplate коду за шаблоном.
- **Що залишати Sonnet:** Прийняття архітектурних рішень, складний рефакторинг, вирішення конфліктів у логіці.

**Патерн "Скрипт-інструмент":** Замість того, щоб намагатися посадити весь Claude Code на локальну модель, викликайте локальну модель як окремий скрипт-інструмент через Bash. Це дає швидкість локалки без втрати якості оркестрації Sonnet.

## Lifecycle hooks субагентів

Окрім PreToolUse/PostToolUse, у субагентів є власні lifecycle hooks:

| Hook | Коли спрацьовує | Типові задачі |
| --- | --- | --- |
| `SubagentStart` | При старті субагента | Rate limiting, logging, валідація вхідного контексту |
| `SubagentStop` | При завершенні субагента | Валідація виводу, data sanitization, performance monitoring |

Субагенти також можуть визначати власні scoped `PreToolUse`/`PostToolUse` hooks — вони застосовуються лише в межах виконання цього субагента.

## Запуск субагентів: технічні вимоги

Координатор запускає субагентів через **Agent tool**. Без явного дозволу спавн фізично неможливий:

```yaml
# .claude/agents/coordinator.md
allowedTools: ["Agent", "Read", "Bash"]
#              ^^^^^^^ обов'язково
```

`Task` — зворотньо-сумісний аліас для `Agent` (SDK v2.1.63 і старіший). Без цього поля у `allowedTools` координатор не може викликати субагентів, навіть якщо промпт явно просить це зробити.

## Вкладені субагенти (субагент викликає субагента)

Субагент, якому дозволено `Agent` tool, може сам породжувати субагентів — утворюючи дерево агентів довільної глибини.

```text
Оркестратор
├── Субагент A  (Agent tool allowed)
│   ├── Субагент A1
│   └── Субагент A2
└── Субагент B
```

**Практичне застосування:** великий план розбивається на підзадачі, кожна підзадача делегується субагенту, який сам паралелізує деталі ще глибше.

**Обмеження, які треба пам'ятати:**

- Кожен рівень стартує з **холодного контексту** — не бачить розмову батька.
- Результат іде **одним повідомленням** назад до батька.
- Глибокі дерева важко налагоджувати і вони накопичують вартість на кожному рівні.
- Щоб субагент міг викликати субагентів, у його `allowedTools` має бути `Agent`.

## Передача контексту між субагентами

Субагенти **не наслідують контекст координатора автоматично**. Кожен виклик повинен містити повний необхідний контекст явно.

Структура контексту, що передається субагенту — два рівні:

| Тип | Поля |
| --- | --- |
| Content | `claim`, `analysis` |
| Metadata | `source_url`, `document_name`, `page_number`, `confidence`, `retrieved_by` |

Приклад передачі фіндингу:

```json
{
  "claim": "Revenue increased 15% YoY",
  "analysis": "Consistent with Q3 trend",
  "source_url": "https://data.internal/snowflake/q4",
  "document_name": "Q4 Financial Report",
  "page_number": 7,
  "confidence": 0.92,
  "retrieved_by": "research_agent_1"
}
```

Без metadata поле `claim` залишається unsourced — синтез-агент не може перевірити або атрибутувати твердження.

Промпт координатора формулюється через **цілі та критерії якості**, а не через покрокові інструкції:

```text
ПОГАНО: "Спочатку прочитай файл A, потім B, потім порівняй"
ДОБРЕ:  "Знайди суперечності між фінансовими даними. Якість: кожне твердження з джерелом і датою."
```

Процедурні кроки обмежують субагента — він не може адаптуватись якщо стан не відповідає скрипту.

## Безпека та Read-only режим

Для `explorer` та `reviewer` ми зазвичай використовуємо **whitelist інструментів**, де відсутні `Edit` та `Write`. Це гарантує, що етап дослідження або перевірки ніколи не призведе до silent-змін у вашому коді.
