## 📝 TL;DR

Планування — це спосіб подолання обмежень AI щодо "втрати контексту" у довгих сесіях. Використання Tasks API та Plan Mode дозволяє розбити складну задачу на верифіковані кроки та зберегти контроль над високоризиковими змінами.

## Навіщо планувати?

LLM-моделі схильні до явища **"Lost in the Middle"** (втрата деталей у середині довгого контексту). Без чіткого плану агент може:

- Забути про необхідність написання тестів.
- Втратити початкову мету після серії виправлень помилок компіляції.
- Виконувати кроки у невірному порядку (наприклад, змінювати код до оновлення схеми БД).

## Рівні декомпозиції

### 1. Tasks API: Тактичне управління

Коли задача займає більше 3-х кроків, агент використовує **Tasks API**. Це не просто список справ, а системний механізм:

- **Персистентність:** Задачі зберігаються в межах поточної сесії та дозволяють агенту не втрачати нитку у довгій роботі. Нова сесія починається з чистого аркуша.
- **Залежності:** Задачі можуть блокувати одна одну (`addBlockedBy`), гарантуючи правильний порядок (Arrange-Act-Assert).
- **Детектор непорозумінь:** Переглядаючи список задач, ви бачите "що саме почув агент" ще до того, як він почне змінювати код.

### 2. Plan Mode: Стратегічний контроль

Це спеціальний режим (`/plan`), де інструменти запису (Write/Edit) заблоковані. Агент зосереджений лише на дослідженні та проєктуванні рішення.

- **GATEKEEPER:** Вихід з режиму планування (`ExitPlanMode`) — це ваш свідомий "Check" рішення.
- **Дослідження без ризику:** У цьому режимі агент може вільно використовувати субагентів для вивчення коду, не вносячи випадкових правок.

## Мапінг складності на підхід

| Складність     | Приклади                         | Рекомендований підхід                                    |
| :------------- | :------------------------------- | :------------------------------------------------------- |
| **Тривіальна** | Javadoc, логування, опечатка     | **Пряме виконання.** Без плану.                          |
| **Середня**    | Логіка в межах одного сервісу    | **Task-список.** Перевірка кроків перед стартом.         |
| **Складна**    | Мультимодульні фічі, нові API    | **Plan Mode.** Обов'язкове design-рев'ю.                 |
| **Критична**   | Flyway, Kafka topics, public API | **Plan Mode + Ризик-чекінг.** Явний апрув кожного кроку. |

## Канонічний робочий цикл

1. **RESEARCH:** Використання субагента [Субагент Explorer](Subagents_Architecture.md) для мапування файлів та потоків даних.
2. **PLAN:** Складання плану в Plan Mode (Tasks API). Опис *як* ми будемо верифікувати кожен крок.
3. **APPROVE:** Валідація плану вами. Коригування порядку або обсягу тестів.
4. **EXECUTE:** Покрокове виконання. Один крок = один зелений тест.
5. **REVIEW:** Фінальна перевірка через субагента [Субагент Reviewer](Subagents_Architecture.md) (відповідність архітектурі та CLAUDE.md).

## Поради для ефективного планування

- **Один крок = один результат:** Уникайте розмитих задач типу "Імплементація сервісу". Краще: "Створення інтерфейсу", "Додавання логіки X", "Покриття тестами Y".
- **Верифікація в плані:** Кожна задача в списку повинна містити спосіб перевірки (наприклад, запуск конкретного тесту).
- **Адаптивність:** Якщо в процесі виконання виявлено нові обставини — зупиніться та оновіть план через `TaskUpdate`.
