## 📝 TL;DR

Шість питань, які варто задати собі перед запуском агента на нетривіальну задачу. Відповідь "не знаю" на будь-яке з них — сигнал зупинитись і прояснити, а не починати реалізацію.

## Коли використовувати

Для тривіальних змін (Javadoc, rename, опечатка) — не потрібно. Для всього іншого — пробіжіться по списку за 2 хвилини. Це дешевше, ніж розгрібати наслідки неправильного старту.

## Чеклист

### ☐ 1. CLAUDE.md актуальний?

Перевірте, що `CLAUDE.md` відображає поточний стан проєкту:

- Команди збірки та тестування актуальні (`./gradlew :module:test`, не застарілий Maven-рецепт).
- Конвенції комітів, найменування гілок, заборонені операції — присутні.
- Немає застарілих правил про модулі, яких вже не існує.

Агент читає `CLAUDE.md` на початку кожної сесії. Застаріла інструкція гірша за відсутню — вона активно вводить в оману.

Детальніше: [Написання CLAUDE.md](CLAUDE_md_Writing_Guide.md)

---

### ☐ 2. `git status` чистий?

```bash
git status
git stash list
```

Незакомічені зміни у робочому дереві — це ризик:

- Агент може читати "напівготові" файли і будувати рішення на їх основі.
- Якщо щось піде не так — складно відрізнити зміни агента від ваших.
- `git diff` після сесії стане нечитабельним.

Якщо є незакомічені зміни — або закомітьте їх, або `git stash` перед стартом.

Також переконайтесь що проєкт компілюється до старту:

```bash
./gradlew compileJava --quiet   # Gradle
mvn compile -q                  # Maven
```

Агент на broken build не може відрізнити свої помилки від вже існуючих — і витратить токени на виправлення чужих регресій.

---

### ☐ 3. Відомо які модулі зачіпає задача?

Запитайте себе: *"Які файли треба змінити?"*

- **Якщо відповідь відома** — вкажіть їх явно у промпті.
- **Якщо ні** — спочатку [субагент Explorer](Explorer_Subagent_Example.md), потім реалізація.

Починати реалізацію без карти змін — це найчастіша причина scope creep: агент сам вирішує що "пов'язане" і змінює зайве.

---

### ☐ 4. Визначено критерій "Done"?

Задача вважається завершеною коли... що саме?

- ✅ *"Зелений тест `PaymentServiceTest#shouldRefundOnTimeout`"* — хороший критерій.
- ✅ *"Full payment module green: `./gradlew :payment-service:test`"* — хороший.
- ❌ *"Код написано"* — поганий: агент зупиниться, не запустивши тести.
- ❌ *"Виглядає правильно"* — суб'єктивно, не верифікується.

Сформулюйте критерій до старту і вкажіть його у промпті: `"Done when: [тест] passes."`

Детальніше: [Ефективні промпти](Effective_Prompting.md), [Верифікаційний цикл](Agentic_Verification_Loop.md)

### ☐ 5. Потрібна Flyway-міграція?

Якщо задача використовує Testcontainers — переконайтесь що Docker запущений перед стартом:

```bash
docker info
```

Якщо задача торкається схеми БД — окремий чеклист:

- Яка остання версія? (`select version from flyway_schema_history order by installed_rank desc limit 1`)
- Наступна міграція буде `VN+1__...sql` — це узгоджено?
- Міграція йде **першим** окремим кроком із verify: `flywayMigrate` без помилок.
- Нова міграція **не модифікує** існуючі файли — тільки новий файл.

Помилка у версії або модифікація існуючої міграції — одна з найбільш болісних помилок у production. [Guardian-хук](Guardian_Hooks.md) ловить деякі з них, але не всі.

---

### ☐ 6. Є ризики для спільних або production-ресурсів?

Задайте собі:

- Зміна торкається публічного API або shared-бібліотеки, яку використовують інші команди?
- Є Kafka-топіки або таблиці, які використовуються в production прямо зараз?
- Зміна вимагає координації з іншою командою (mobile freeze, release branch)?

Якщо хоча б одна відповідь — "так":

1. Використайте [Plan Mode](Planning_and_Tasks.md) з явним апрувом кожного кроку.
2. Додайте `think hard` для архітектурного аналізу ризиків.
3. Фінальний diff — обов'язково через [субагент reviewer](Reviewer_Subagent_Example.md) з явним переліком що перевіряти.

## Швидка версія (для сканування)

```text
☐ CLAUDE.md актуальний (команди, конвенції, немає застарілих правил)
☐ git status чистий (або stash)
☐ Відомі файли для зміни (якщо ні — Explorer першим)
☐ Критерій Done сформульований (зелений тест, не "код написано")
☐ Flyway: версія наступної міграції відома, вона йде першим кроком
☐ Ризики production/shared API: якщо є — Plan Mode + think hard + reviewer
```
