## 📝 TL;DR

Динамічна оркестрація — це коли оркестратор вирішує наступний крок на основі результату попереднього, а не слідує заздалегідь прописаній послідовності. Замість фіксованого ланцюжка `Explorer → Test-runner → Reviewer` агент сам розгалужує, повторює або паралелізує задачі залежно від того, що знайшов.

> Не плутати з офіційною фічею Claude Code [Dynamic Workflows](Claude_Code_Dynamic_Workflows.md) — де Claude генерує JS-скрипт оркестрації для запуску сотень субагентів поза сесією. Ця нотатка — про адаптивну логіку всередині сесії.

## Static vs Dynamic оркестрація

| Static workflow | Dynamic orchestration |
| :--- | :--- |
| Послідовність кроків відома заздалегідь | Наступний крок залежить від поточних результатів |
| Ваш код або промпт диктує кожен крок | Оркестратор вирішує сам |
| Передбачувано, дешево | Гнучко, але дорожче та важче налагоджувати |
| Підходить для рутинних задач | Підходить для дослідницьких задач |

## Ключові патерни

### 1. Conditional branching — розгалуження за результатом

Оркестратор аналізує результат і обирає різні шляхи:

```text
Explorer знайшов Kafka-консьюмерів?
  → Так: запустити Kafka-специфічний аналізатор
  → Ні:  перейти одразу до Test-runner
```

В промпті оркестратора це виглядає як інструкція з умовою:

```text
After exploration, if Kafka consumers are found — run the kafka-analyzer agent.
Otherwise proceed directly to test-runner.
```

### 2. Retry loop — повтор до умови

Агент виконує дію і перевіряє результат у циклі:

```text
Test-runner запустив тести
  → Тести впали: запустити діагностичний агент, виправити, повторити
  → Тести зелені: передати Reviewer
```

Щоб уникнути нескінченного циклу — завжди задавайте явний ліміт:

```text
Retry up to 3 times. If tests still fail after 3 attempts — stop and report.
```

### 3. Adaptive parallelism — паралелізм за потребою

Оркестратор вирішує паралелізувати тільки якщо знайдено кілька незалежних об'єктів:

```text
Explorer знайшов 4 незалежні модулі
  → Запустити 4 Explorer-субагенти паралельно (по одному на модуль)
Explorer знайшов 1 великий файл
  → Запустити 1 субагент послідовно
```

## Практичний приклад: аудит Spring Boot проєкту

```text
1. Explorer досліджує структуру проєкту
        ↓
2. Оркестратор аналізує результат:
   ├── Знайдено Kafka consumers?  → запустити kafka-analyzer
   ├── Знайдено складні JPA queries? → запустити jpa-analyzer
   └── Знайдено нові REST endpoints? → запустити api-reviewer
        ↓
3. Всі аналізатори завершили роботу (паралельно, якщо незалежні)
        ↓
4. Test-runner запускає тести
   └── Впали? → діагностика → виправлення → повтор (max 3)
        ↓
5. Reviewer виносить вердикт
```

Жоден з кроків 2–3 не визначений заздалегідь — оркестратор вирішує їх у реальному часі.

## Як задати правила оркестратору

Ключ — system prompt або CLAUDE.md з умовними інструкціями:

```markdown
## Orchestration rules

After exploration:
- If Kafka consumers found → run kafka-analyzer agent before proceeding.
- If 3+ independent modules found → analyze them in parallel.

On test failure:
- Diagnose, fix, and retry. Max 3 retries.
- If still failing after 3 attempts → stop and explain why.
```

## Коли NOT використовувати

- **Рутинна задача з відомою послідовністю** — статичний workflow дешевший і передбачуваніший.
- **Коли помилка дорога** — динамічне розгалуження важко аудитувати. Для критичних змін (міграції БД, деплой) краще явний покроковий контроль.
- **Без ліміту ітерацій** — retry loop без обмежень може витратити весь бюджет токенів.
