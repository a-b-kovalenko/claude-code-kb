## 📝 TL;DR

ADR (Architecture Decision Record) — короткий документ, що фіксує важливе архітектурне рішення: контекст, само рішення, розглянуті альтернативи та наслідки. Відповідає на питання "чому", а не "що".

---

## Навіщо ADR

Код показує *що* побудовано. Git-історія показує *як* змінювалось. ADR пояснює *чому* обрано саме цей підхід — і чому відкинуто альтернативи.

Без цього наступний розробник (або ви самі через рік) може переглянути рішення, не знаючи контексту. Розділ "Alternatives Considered" — найцінніший: захищає від повторного повернення до вже відхиленого варіанту.

---

## Структура файлу

### Title

Коротка назва рішення (стає частиною імені файлу).

### Status

Поточний стан: `Proposed` → `Accepted` → `Deprecated` або `Superseded by ADR-XXXX`.

### Context

Що відбувається, які обмеження та вимоги спонукали прийняти рішення.

### Decision

Що саме вирішено і чому.

### Alternatives Considered

Які альтернативи розглядались і чому їх відкинуто.

### Consequences

Наслідки рішення: позитивні, негативні, trade-offs.

## Нумерація та іменування

```text
docs/adr/
  README.md
  0001-project-setup.md
  0002-use-postgresql.md
  0003-event-sourcing-for-orders.md
```

- Чотиризначний номер із нуля: `0001`, `0002`…
- Коротка назва через дефіс, lowercase
- Перший ADR у проєкті: `0001-project-setup.md` — документує рішення прийняти цей підхід

## Шаблон

```markdown
# ADR-XXXX: [Назва рішення]

## Status

Proposed

## Context

[Опис контексту, вимог та обмежень]

## Decision

[Що вирішено і чому]

## Alternatives Considered

| Альтернатива | Причина відхилення |
|---|---|
| [варіант A] | [чому ні] |
| [варіант B] | [чому ні] |

## Consequences

- ✅ [позитивний наслідок]
- ⚠️ [trade-off або ризик]
```

## Приклад

```markdown
# ADR-0002: Вибір PostgreSQL як основної БД

## Status

Accepted

## Context

Потрібна реляційна БД з підтримкою JSONB для метаданих та повнотекстового пошуку.

## Decision

Обрано PostgreSQL 16: підтримує JSONB, повнотекстовий пошук та партиціювання з коробки.

## Alternatives Considered

| Альтернатива | Причина відхилення |
|---|---|
| MySQL 8 | Слабша підтримка JSONB, немає партиціювання за range |
| MongoDB | Команда не має досвіду, складніше транзакційне управління |

## Consequences

- ✅ Повнотекстовий пошук без зовнішніх сервісів
- ✅ Команда має досвід з PostgreSQL
- ⚠️ Потребує Testcontainers для інтеграційних тестів
```

## Інтеграція з Antora

Якщо проєкт використовує Antora, ADR кладуться в:

```text
docs/modules/ROOT/pages/adr/
  0001-project-setup.adoc
  0002-use-postgresql.adoc
```

Можна зберігати як `.md` і підключати через `supplemental-ui` — залежить від конфігурації Antora.

---

## Роль Claude Code

Агент може самостійно пропонувати написати ADR після архітектурної дискусії:

> *"Ми щойно вирішили перейти на event sourcing. Записати це як ADR-0003?"*

Для цього варто вказати у `CLAUDE.md` проєкту:

> Після архітектурних рішень пропонуй написати ADR у `docs/adr/`.
