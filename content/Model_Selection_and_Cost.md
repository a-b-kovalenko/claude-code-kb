## 📝 TL;DR

Вибір моделі — це баланс між якістю, швидкістю і вартістю. Haiku для механічної роботи (пошук, читання файлів), Sonnet для синтезу та коду, Opus — лише для архітектурних рішень де якість критична. Неправильний вибір не ламає результат, але коштує часу або грошей.

## Три моделі і їх ніші

| Модель | ID | Характеристика |
| :--- | :--- | :--- |
| **Haiku 4.5** | `claude-haiku-4-5-20251001` | Найшвидший, найдешевший. Відмінний для читання, пошуку, класифікації |
| **Sonnet 4.6** | `claude-sonnet-4-6` | Основна робоча модель. Баланс якості і швидкості |
| **Opus 4.8** | `claude-opus-4-8` | Максимальна глибина. Для задач де якість рішення критична |

Claude Code за замовчуванням використовує Sonnet. Зміна моделі не потребує коду — тільки конфігурація.

## Матриця: задача → модель

| Задача | Модель | Чому |
| :--- | :--- | :--- |
| Пошук файлів, grep, навігація | Haiku | Механічна робота, не потребує міркувань |
| Читання та індексування коду | Haiku | Обсяг важливіший за глибину |
| Аналіз стектрейсів і логів | Sonnet | Потрібне розуміння контексту |
| Написання і рефакторинг коду | Sonnet | Основна задача, потрібна якість |
| Code review, вердикт APPROVE/REJECT | Sonnet | Синтез правил + diff |
| Архітектурне рішення з компромісами | Opus / Sonnet + `/think` | Потрібне глибоке міркування |
| Діагностика race condition чи дедлоку | Opus / Sonnet + `/think` | Нетривіальна логіка |
| Генерація boilerplate, Javadoc | Haiku | Шаблонна робота |

## Налаштування моделі

### За замовчуванням для проєкту (`settings.json`)

```json
{
  "model": "claude-sonnet-4-6"
}
```

### Для конкретного субагента (`agents/explorer.md`)

```yaml
---
name: explorer
model: claude-haiku-4-5-20251001
---
```

### Для всіх субагентів одразу (змінна середовища)

```json
{
  "env": {
    "CLAUDE_CODE_SUBAGENT_MODEL": "claude-haiku-4-5-20251001"
  }
}
```

Субагент може перевизначити це власним `model:` у frontmatter. Якщо поле відсутнє — субагент успадковує (`inherit`) модель оркестратора.

## Стратегії економії

**Haiku для Explorer** — найбільший виграш. Explorer читає десятки файлів і повертає структурований список. Це не потребує Sonnet — Haiku справляється швидше і дешевше, не жертвуючи якістю навігації.

**Sonnet лише для синтезу** — оркестратор на Sonnet отримує готові звіти від Haiku-субагентів і синтезує рішення. Важка робота вже зроблена дешево.

**Уникайте зайвого читання файлів** — кожен `Read` займає токени. Перед читанням великого файлу перевірте чи він справді потрібен: `grep` або `Glob` часто дають відповідь без повного читання.

**`/compact` перед новим блоком задачі** — стиснення контексту зменшує розмір "хвоста" розмови, який модель несе з собою на кожному кроці.

**Паралельні Haiku-агенти замість одного Sonnet** — три Haiku-агенти, що паралельно досліджують три модулі, часто швидші і дешевші за один Sonnet, що робить те саме послідовно.

## Extended Thinking: коли окупається

`/think` чи `ultrathink` збільшують витрати токенів, але не завжди дають кращий результат. Окупається коли:

- Кілька рівнозначних архітектурних варіантів без очевидного переможця.
- Нетривіальний баг (race condition, дедлок, непередбачувана поведінка Kafka).
- Рішення з довготривалими наслідками (схема БД, API-контракт).

Не окупається для:

- Написання тестів за шаблоном.
- Рефакторинг з чіткими правилами.
- Будь-яка задача де ви самі знаєте відповідь за 10 секунд.

Детальніше: [🧠 Extended Thinking та /think](Extended_Thinking.md).

## Підписка vs API

**Claude Pro/Team/Enterprise (підписка):** вартість фіксована. Вибір моделі впливає на **швидкість** і **якість**, але не на рахунок. Haiku тут цінний за швидкість, а не за економію грошей.

**Anthropic API (pay-per-token):** вибір моделі прямо впливає на рахунок. Haiku значно дешевший за токен ніж Sonnet — зниження вартості в рази для задач де він підходить.
