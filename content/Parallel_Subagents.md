## 📝 TL;DR

Паралельні субагенти — це кілька `Task`-викликів в одному повідомленні оркестратора. Кожен субагент отримує власний ізольований контекст і виконується одночасно; оркестратор чекає на всі результати і синтезує їх. Виграш у часі пропорційний кількості незалежних гілок.

## Зв'язок з `.claude/agents/`

Паралельність і визначення агентів — ортогональні речі:

- **`.claude/agents/*.md`** описує *що* є агент: роль, модель, whitelist інструментів. Це опціональна заготовка.
- **`Task`-виклик** визначає *як* агент запускається — з посиланням на іменованого агента або з inline-інструкцією.

Оркестратор може запускати паралельно як іменованих агентів (`"Use the explorer agent"`), так і ad-hoc задачі без жодного файлу в `/agents`. Файли в `.claude/agents/` дають перевагу — зафіксована модель, обмежені інструменти, готовий system prompt — але не є обов'язковими.

## Як це працює

Коли головний агент робить кілька `Task`-викликів **в одній відповіді** — Claude Code запускає їх паралельно:

```text
Оркестратор
├── Task → Explorer (domain/)       ┐
├── Task → Explorer (application/)  ├─ виконуються одночасно
└── Task → Explorer (infrastructure/)┘
         ↓
   Оркестратор отримує три звіти і синтезує
```

Якщо `Task`-виклики у **різних** повідомленнях — вони послідовні.

## Коли варто запускати паралельно

Ключове питання: **чи залежить задача B від результату задачі A?**

| Сценарій | Паралельно? |
| :--- | :--- |
| Аналіз трьох незалежних модулів | Так |
| Перевірка міграцій і тестів одночасно | Так |
| Explorer → Test-runner (тести залежать від знайдених файлів) | Ні (послідовно) |
| Test-runner → Reviewer (рев'ю залежить від результатів тестів) | Ні (послідовно) |

## Як оркестратор вирішує запустити паралельно

Агент не визначає це автоматично. Є три механізми:

1. **Явна інструкція від користувача** — найнадійніше:
   > *"Паралельно досліди модулі `order`, `payment` і `notification`."*
   Агент отримує пряму команду і робить кілька `Task` в одній відповіді.

2. **Правило в system prompt агента або CLAUDE.md:**

   ```text
   When research tasks are independent, always spawn them in parallel.
   ```

   Оркестратор сам обирає паралельність при плануванні.

3. **Власне міркування агента** — найменш надійне. Без явного вказання агент може обрати послідовний підхід за замовчуванням.

**Практичне правило:** якщо паралельність важлива — просіть явно. Покладатися на те, що агент сам здогадається, ризиковано.

## Практичний приклад: аналіз Spring Boot проєкту

Замість послідовного обходу модулів:

> *"Запусти три Explorer-субагенти паралельно: один досліджує `order/`, другий — `payment/`, третій — `notification/`. Зведи результати у єдину карту залежностей."*

Оркестратор запустить три `Task` одночасно. Кожен Explorer поверне список ключових класів та залежностей свого модуля — без взаємного блокування.

Інший приклад — паралельна перевірка:

> *"Одночасно: (1) запусти тести модуля `order`, (2) перевір Flyway-міграції на наявність конфліктів версій."*

Два незалежних завдання — два субагенти — половина часу.

## Обмеження

- **Запис у спільні файли** — якщо обидва субагенти редагують один файл, виникне конфлікт. Паралельні субагенти мають бути read-only або працювати з різними файлами.
- **Спільний стан** — субагенти не бачать результатів один одного в реальному часі. Вся комунікація — через оркестратора після завершення.
- **Вартість** — паралельні субагенти споживають токени одночасно. Для дрібних задач послідовний підхід може бути дешевшим.
