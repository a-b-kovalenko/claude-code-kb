## 📝 TL;DR

Worktree isolation — параметр `isolation: "worktree"` в Agent-виклику, що запускає субагента у відгалуженій копії репозиторію. Основний working tree залишається незайманим. Якщо агент нічого не змінив — worktree прибирається автоматично; якщо змінив — повертає шлях і назву гілки для ревʼю і злиття.

## Що таке git worktree

Git дозволяє тримати кілька робочих директорій з одного репозиторію одночасно — кожна на своїй гілці. `isolation: "worktree"` використовує цей механізм: агент отримує ізольовану копію файлової системи проєкту і не може зіпсувати ваш поточний стан роботи.

## Як запустити

`isolation: "worktree"` — це параметр, який **головний агент (Claude) передає сам** при виклику інструменту `Agent`. Ти не пишеш цей код у конфігах — достатньо попросити явно в запиті.

Достатньо попросити словами:

```text
"Зроби великий рефакторинг PaymentService, використай worktree isolation"
```

Claude підставить параметр самостійно:

```text
Agent({
  description: "Large-scale refactor of payment module",
  isolation: "worktree",
  prompt: "Refactor PaymentService to use the new event-driven pattern..."
})
```

## Lifecycle

```text
1. Claude Code створює git worktree у тимчасовій директорії
         ↓
2. Субагент виконує задачу в ізоляції
         ↓
3a. Агент не зробив змін → worktree прибирається автоматично
3b. Агент зробив зміни  → повертається { path, branch }
         ↓
4. Ви ревʼюєте diff, мерджите або відкидаєте
```

Результат при наявності змін:

```text
Worktree created at: /private/var/folders/.../worktree-abc123
Branch: worktree/refactor-payment-1718123456
```

## Коли використовувати

| Сценарій | Чому worktree |
| :--- | :--- |
| Великий рефакторинг (10+ файлів) | Агент може зламати щось несподіване — ізоляція дає час на ревʼю |
| Паралельне дослідження двох підходів | Два агенти в різних worktree → порівнюєте diff і обираєте |
| Ризикована зміна (міграція БД, API-брейкінг) | Зміни не потрапляють у ваш working tree без явного злиття |
| Автономна задача без нагляду | Агент працює поки ви займаєтесь іншим — жодного ризику для поточного стану |

## Коли НЕ потрібно

- Дрібні зміни (один файл, явний локальний контекст) — overhead без користі.
- Дослідницькі задачі (агент тільки читає код і пояснює) — worktree потрібен лише коли агент щось записує.
- Коли зміни мають бути негайно видимі в основному дереві (наприклад, генерація конфігу для подальшої роботи).
- Якщо репозиторій не ініціалізований git — worktree вимагає git.

## Нюанси для Java-проєктів

**IntelliJ IDEA і індексування.** Worktree — тимчасова папка поза проєктом. Якщо відкрити її в IDEA, запуститься повне індексування (5–10 хв на великому проєкті). Краще робити ревʼю через `git diff` або `gh pr checkout` у поточному проєкті — тоді IDEA вже проіндексована.

**Gradle build-кеш.** `~/.m2` (Maven) — глобальний, worktree його не дублює. Але Gradle тримає `build/` окремо для кожної директорії, тому якщо агент запускає тести — буде окремий cold build. Для важких тестів worktree краще використовувати без запуску build-агентом, або налаштувати `--build-cache`.

**Редагуєш той самий модуль — буде конфлікт.** Якщо агент рефакторить модуль X у worktree, а ти паралельно чіпаєш ті ж файли в основному дереві — мердж буде болючим. Правило: поки агент працює в ізоляції над модулем X — не чіпай модуль X у main.

## Практики

**Проси явно, не автоматизуй.** Критерій "великий рефакторинг" суб'єктивний — якщо вбити його в CLAUDE.md, агент застосовуватиме ізоляцію там, де вона не потрібна. Краще просити вручну, коли сам усвідомлюєш ризик.

**Паралельне дослідження — найкорисніший сценарій.** Два агенти, два підходи, один `git diff` — і ти обираєш кращий. Це рідко використовують, але це реальна перевага worktree, а не тільки захист від помилок.

**Diff перед злиттям — обов'язок, не опція.** Агент міг зробити технічно правильні зміни, які не відповідають твоєму наміру. `git diff main...branch` перед `git merge` — завжди.

**Прибирай гілки після злиття.** Worktree-гілки не видаляються автоматично після злиття — вони накопичуються. `git branch -d worktree/...` одразу після merge.

## Злиття результату

Після того як агент повернув `{ path, branch }`:

```bash
# Переглянути що зробив агент
git diff main...worktree/refactor-payment-1718123456

# Злити якщо задоволені
git merge worktree/refactor-payment-1718123456

# Або відкрити PR через gh
gh pr create --head worktree/refactor-payment-1718123456
```

Worktree-гілку можна видалити після злиття:

```bash
git branch -d worktree/refactor-payment-1718123456
```

Масове очищення всіх змерджених worktree-гілок:

```bash
git branch --merged main | grep 'worktree/' | xargs git branch -d
```
