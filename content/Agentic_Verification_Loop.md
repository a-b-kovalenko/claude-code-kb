## 📝 TL;DR

Верифікаційний цикл — це здатність AI-агента самостійно та швидко перевіряти свої зміни. Це фундамент якості, який базується на гранулярному тестуванні, перевикористанні контейнерів та чітких критеріях завершення задачі.

## Чому верифікація є критичною?

На відміну від людини, AI-агент може робити сотні спроб редагування коду. Якщо кожна спроба вимагає 10-хвилинного очікування на повну збірку проєкту, розробка стає неефективною. Якість агентської роботи прямо залежить від **вартості та швидкості зворотного зв'язку**.

## Ключові принципи швидкого циклу

### 1. Гранулярність замість монолітності

Агент ніколи не повинен запускати повний білд проєкту (`mvn install` або `./gradlew build`) для перевірки локальної зміни.

- **Gradle:** Використовуйте `:module:test` для ізольованого тестування модуля.
- **Maven:** Використовуйте `-pl (project list)` для фокусування лише на зміненому коді.
Це скорочує час очікування з хвилин до секунд.

### 2. Оптимізація інфраструктури (Testcontainers)

Ми використовуємо реальні бази даних та брокери через Testcontainers, але робимо це розумно:

- **Reuse Mode:** Увімкнення `testcontainers.reuse.enable=true` у глобальних налаштуваннях дозволяє контейнерам залишатися активними між запусками тестів. Агент не витрачає час на "холодний старт" Docker-контейнерів при кожній правці коду.
- **Dynamic Property Source:** Автоматичне підключення до піднятих контейнерів без потреби в ручному налаштуванні `application-test.yml`.

### 3. Автономна Dev-інфраструктура

Для складних сценаріїв, де тести не покривають усе, агент повинен мати можливість підняти локальне оточення через `docker compose`. Це дозволяє йому:

- Проводити "димові" (smoke) тести.
- Перевіряти міграції Flyway на реальних даних.
- Інспектувати стан Kafka-топіків.

### 4. Машинно-перевірюваний критерій Done

Задача не вважається завершеною, якщо агент просто "написав код". Критерій готовності — це **зелений колір конкретного тесту**.

- Агент повинен спершу відтворити баг через тест, що падає.
- Виконати правку.
- Отримати підтвердження проходження тесту від субагента `test-runner`.

## Роль субагента Test-runner

Для підтримки чистоти основного контексту, запуск тестів та аналіз логів помилок варто делегувати субагенту `test-runner`. Це дозволяє головному агенту не засмічувати свою пам'ять тисячами рядків стектрейсів, отримуючи лише фінальний звіт про успіх або причину падіння.
