## 📝 TL;DR

Claude Code SDK — програмний інтерфейс для запуску Claude Code з коду, а не з терміналу. Доступний через CLI, TypeScript і Python; підтримує ті самі інструменти, що й інтерактивна сесія. Для архітектора команди це **API-поверхня для вбудовування AI у SDLC**: quality gates, автоматизовані рев'ю, командні хуки — все це будується поверх SDK.

## Доступні інтерфейси

| Інтерфейс | Коли використовувати |
| :--- | :--- |
| CLI (`claude -p`) | Shell-скрипти, швидкі одноразові запити |
| TypeScript SDK | Хуки, Node.js-інструменти, складніша логіка |
| Python SDK | Data-pipeline, Jupyter, ML-workflow |

Усі три мають ті самі можливості — різниця лише в зручності інтеграції з конкретним стеком.

## Дозволи за замовчуванням

SDK стартує в **read-only** режимі: дозволені читання файлів, пошук (grep), обхід директорій. Запис файлів, запуск команд — заблоковані.

Щоб увімкнути запис, передайте `allowTools` у параметрах запиту:

```typescript
const result = await claude.query({
  prompt: "...",
  options: {
    allowTools: ["edit", "write", "bash"]
  }
});
```

Альтернатива — налаштувати дозволи через `.claude/settings.json` у директорії проєкту.

## Формат виводу

SDK повертає **сирий діалог** між локальним Claude Code і моделлю — повідомлення за повідомленням. Фінальна відповідь — останнє повідомлення у списку.

Це корисно для діагностики: видно, які інструменти викликались, які файли читались, де агент змінив підхід.

## Кут зору архітектора команди

SDK — це не інструмент окремого розробника, а **інфраструктурний шар** для команди. Архітектор будує його один раз; вся команда отримує AI-підсилені процеси без необхідності розуміти деталі Claude API.

### Типові точки інтеграції у командний SDLC

**Quality gates у CI/CD**
Замість того щоб кожен розробник запускав Claude вручну — зробити це обов'язковим кроком пайплайну:

```bash
# GitHub Actions: Claude рев'ює PR перед merge
claude -p "Review this PR for architecture compliance:
- no business logic in controllers
- all new services follow hexagonal architecture
- Flyway migrations are safe for production" \
  --output-format json
```

**Командні хуки (спільний .claude/)**
Хуки у `.claude/hooks/` — це командна інфраструктура. Архітектор пише їх один раз і комітить у репо; кожен розробник отримує автоматичні перевірки:

- Flyway-guard: блокує небезпечні міграції до запуску
- Architecture guard: Claude через SDK перевіряє, що новий сервіс не порушує шарування
- Duplication detector: secondary Claude шукає дублікати в `repository/`

**Стандартизація AI-взаємодій**
SDK дозволяє загорнути Claude у внутрішній CLI-інструмент команди з фіксованими промптами, контекстом і дозволами — розробники отримують зручний інтерфейс без доступу до сирого API.

### Що SDK дає, чого немає в інтерактивній сесії

| Потреба | Інтерактивна сесія | SDK |
| :--- | :--- | :--- |
| Автозапуск у CI | ні | так |
| Програмний контроль дозволів | ні | так (`allowTools`) |
| Паралельний запуск агентів | обмежено | так |
| Інтеграція у командний тулінг | ні | так |
| Детермінований аудит (хто що запитав) | ні | так (сирий діалог) |

### Для Java-команди зокрема

SDK стає актуальним не для написання Java-коду, а для **метапроцесів навколо нього**. Архітектор пише хуки і CI-скрипти на bash/TypeScript/Python — вони працюють з Java-проєктом через Claude. Самі розробники в Java-коді SDK не бачать.

## Основний сценарій використання

SDK не призначений для standalone-застосунків. Його ніша — **допоміжні ролі всередині існуючих проєктів**:

- **Хуки** — запустити окремий інстанс Claude для перевірки змін (наприклад, виявлення дублювання коду перед коммітом).
- **Скрипти** — автоматизувати повторювані задачі з логікою, яка потребує розуміння коду.
- **Pipeline** — додати intelligence до існуючого CI/CD або dev-workflow.
