## 📝 TL;DR

`/batch` — скіл для масштабних паралельних змін у кодовій базі. Сам декомпозує задачу на незалежні одиниці, показує план, після апруву запускає окремий субагент у ізольованому git worktree для кожної частини — і відкриває окремий PR.

## Синтаксис

```text
/batch <інструкція>
```

Інструкція — вільна мова. Скіл сам досліджує кодову базу і визначає межі одиниць роботи.

```text
/batch migrate src/ from Solid to React
/batch add @Transactional to all service methods in src/services/
/batch add OpenAPI annotations to all REST controllers
```

## Як формулювати інструкцію

Інструкція працює краще коли містить три елементи: **дія + що змінити + де**.

| Елемент | Приклад |
| :--- | :--- |
| Дія | `migrate`, `add`, `replace`, `refactor`, `extract` |
| Що | `@Transactional`, `OpenAPI annotations`, `Lombok → records` |
| Де | `src/services/`, `all REST controllers`, `src/` |

**Хороші інструкції:**

```text
/batch replace manual null checks with Optional in src/domain/
/batch extract magic numbers to constants in src/
/batch add @Slf4j and replace System.out.println in src/services/
```

**Погані інструкції:** надто розмиті без scope — скіл може неправильно визначити межі одиниць:

```text
/batch improve code quality
/batch refactor the project
```

Специфічні патерни, винятки і конвенції краще описати у `CLAUDE.md` — усі субагенти його читають.

## Як працює

1. Досліджує кодову базу
2. Декомпозує задачу на **5–30 незалежних одиниць**
3. Показує план — чекає апруву
4. Запускає **окремий background субагент у ізольованому git worktree** для кожної одиниці
5. Кожен субагент реалізує свою частину → запускає тести → відкриває PR

Потребує git-репозиторій (worktree-ізоляція обов'язкова).

## Апрув плану

План показується у поточній розмові — це звичайна відповідь у сесії, не окремий файл. Апрув — твоя відповідь у тій самій сесії.

Поки сесія відкрита — можна читати план скільки завгодно і потім сказати "виконуй". Якщо закрити термінал між переглядом і апрувом — план зникне разом з контекстом. Відновлення через [`/resume`](../Commands_Reference.md) технічно можливе, але чи підхопить `/batch` стан "чекаю апруву" — невідомо; швидше за все доведеться запускати заново.

Якщо хочеш зберегти план — скопіюй вручну у `.workspace/` до апруву.

## Окремі PR на кожну одиницю

Кожен субагент відкриває свій PR — якщо задача декомпозована на 20 одиниць, отримаєш 20 PR. Так відбувається тому що задачі ізольовані у різних worktrees і не мають спільного diff.

Переваги:

- Можна мерджити по одному, не чекаючи всіх
- Відхилений PR не блокує решту
- Кожен PR маленький і фокусований — зручно ревʼювати

Для Java-проєкту це може виглядати як 15 PR — по одному на кожен сервіс після `/batch add @Transactional to all service methods`.

**Ризик конфліктів:** якщо кілька одиниць чіпають один файл або shared код — конфлікти при мерджі неминучі. `/batch` намагається декомпозувати на незалежні одиниці, але це Claude's judgment без гарантій. Щоб зменшити ризик:

- Формулюй scope по ізольованих модулях (`src/services/`, `src/controllers/`) — не по всьому `src/`
- Уникай задач що торкаються shared коду (базові класи, конфігурація) — їх краще робити окремою сесією
- Мерджи PR послідовно, перевіряючи конфлікти перед наступним

## Контекст для субагентів

Кожен субагент читає `CLAUDE.md` проєкту. Якщо задача потребує специфічних патернів, конвенцій або винятків — краще прописати їх у `CLAUDE.md` заздалегідь, ніж деталізувати в інструкції до `/batch`.
