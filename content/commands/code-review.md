## 📝 TL;DR

`/code-review` — ревʼю поточного diff на баги і cleanup. Рівень деталізації задається аргументом (`low` → `ultra`). Знахідки можна застосувати (`--fix`) або запостити як коментарі до PR (`--comment`). Тільки cleanup без багів — `/simplify`.

## Синтаксис

```text
/code-review
/code-review [рівень]
/code-review [рівень] [--fix] [--comment] [target]
/code-review ultra <PR#>
```

**`target`** — шлях до файлу, директорії або PR-посилання. Без target — поточний uncommitted diff.

## Рівні деталізації

| Рівень | Характер знахідок |
| :--- | :--- |
| `low` | Мінімум — тільки очевидні, високодостовірні баги |
| `medium` | Баланс між покриттям і точністю (за замовчуванням) |
| `high` | Ширше покриття, можуть бути менш певні знахідки |
| `xhigh` | Ще ширше — для критичного коду |
| `max` | Максимальне покриття |
| `ultra` | Глибоке хмарне ревʼю кількома агентами паралельно |

## Прапори

**`--fix`** — застосувати знахідки до working tree автоматично.

**`--comment`** — запостити знахідки як inline-коментарі до GitHub PR (потребує `gh` CLI).

Обидва прапори можна комбінувати з будь-яким рівнем.

## Типові сценарії

```text
/code-review               ← швидка перевірка перед комітом
/code-review high          ← детальніше, перед відкриттям PR
/code-review --fix         ← перевірити і одразу виправити
/code-review ultra 42      ← хмарне ревʼю PR #42
/code-review --comment     ← додати коментарі до відкритого PR
```

## Порівняння з `/simplify`

| | `/code-review` | `/simplify` |
| :--- | :--- | :--- |
| Баги | Шукає | Не шукає |
| Cleanup | Так, з `--fix` | Так, автоматично |
| Вибір рівня | Так | Ні (завжди 4 субагенти) |
| Коли | Перед merge або аудит | Після реалізації, перед PR |
