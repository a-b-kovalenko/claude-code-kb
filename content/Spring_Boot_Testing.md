## 📝 TL;DR

Ми будуємо тестування навколо реального оточення (Testcontainers), відмовляючись від H2 на користь повної сумісності з Postgres/Kafka. Наші інструменти: JUnit 5 для структури, AssertJ для перевірок, DbRider для даних та Awaitility для асинхронності.

## Чому реальне оточення (Testcontainers)?

Використання H2 для тестів Postgres-проєкту — це антипатерн. Вона приховує баги специфічних діалектів, не підтримує JSONB, складні індекси та обмеження (constraints).

- **Правило:** Тести мають бігти на тій же версії бази, що і Prod.
- **Оптимізація:** Використовуйте `testcontainers.reuse.enable=true`, щоб не чекати підняття контейнерів при кожному запуску.

## Ефективне використання тестових зрізів (Slices)

Не кожному тесту потрібен повний `@SpringBootTest`. Використовуйте вузькі зрізи для швидкості:

| Зріз | Що тестуємо | Переваги |
| :--- | :--- | :--- |
| **Plain JUnit 5** | Бізнес-логіка (POJO) | Найшвидший. Жодного Spring Context. |
| **`@DataJpaTest`** | Репозиторії, SQL, Mapping | Тільки JPA компоненти + DB. |
| **`@WebMvcTest`** | Контролери, серіалізація | Тільки Web-шар. Сервіси мокаються. |
| **`@SpringBootTest`** | Повний integration slice | Весь контекст. Використовувати тільки коли треба перевірити взаємодію багатьох сервісів. |

## Мистецтво перевірок з AssertJ

Відмовтеся від `assertEquals`. Використовуйте Fluent API AssertJ для створення читабельних перевірок:

```java
assertThat(userResponse)
    .isNotNull()
    .extracting(UserDto::getRole, UserDto::getStatus)
    .containsExactly(Role.ADMIN, Status.ACTIVE);

assertThat(list)
    .hasSize(3)
    .allSatisfy(item -> assertThat(item.getPrice()).isPositive());
```

## DbRider: Керування станом БД

Для інтеграційних тестів ми використовуємо YAML-датасети замість ручного створення сутностей у коді.

- **`@DataSet`**: Підготовка даних. Завжди використовуйте `cleanBefore = true`, щоб уникнути впливу попередніх тестів.
- **`@ExpectedDataSet`**: Перевірка фінального стану БД після виконання дії.
- **Уникайте `@Transactional`**: Не використовуйте цю анотацію на методах тестів для "відкату" змін. Це маскує помилки `LazyInitializationException` та проблеми з commit-багами.

## Асинхронність та Kafka

Тестування Kafka — це завжди тестування побічних ефектів.

- **Awaitility**: Ніколи не використовуйте `Thread.sleep()`. Використовуйте асинхронне очікування умови:

```java
await()
    .atMost(5, SECONDS)
    .untilAsserted(() -> {
        var event = repository.findById(id);
        assertThat(event).isPresent();
    });
```

- **Error Paths**: Обов'язково тестуйте негативні сценарії — що стається, коли десеріалізація Kafka-події падає (чи йде вона в DLQ?).

## Структура тесту (AAA Pattern)

Кожен тест повинен бути візуально розділений порожніми рядками на три блоки:

1. **Arrange**: Підготовка (моки, дані).
2. **Act**: Виконання однієї конкретної дії.
3. **Assert**: Перевірка результату.
