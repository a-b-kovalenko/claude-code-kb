## 📝 TL;DR

`md-lint.sh` — PostToolUse-хук, що запускає `markdownlint` після кожної зміни `.md`-файлу **у межах проєкту** і блокує агента (exit 2), якщо знайдено порушення. Замість "попереджень, які ніхто не виправляє" — примусове виправлення.

## Для чого потрібен

Без хука агент може писати markdown з порушеннями: відсутні порожні рядки навколо заголовків, code-блоки без вказівки мови, неправильне форматування таблиць. Інструкція в `CLAUDE.md` не гарантує дотримання. Хук гарантує.

## Конфігурація лінтера

`.markdownlint.json` у корені vault:

```json
{
  "MD013": false,
  "MD041": false
}
```

| Правило | Причина відключення |
| :--- | :--- |
| MD013 (line-length) | Нотатки — проза, довгі рядки є нормою |
| MD041 (first-line-h1) | Vault-конвенція: перший рядок — nav-лінк, не H1 |

Усі інші правила активні: MD022 (пробіли навколо заголовків), MD032 (пробіли навколо списків), MD040 (мова code-блоків), MD060 (форматування таблиць).

## Скрипт

`.claude/hooks/md-lint.sh`:

```bash
#!/usr/bin/env bash

input=$(cat)
file=$(printf '%s' "$input" | python3 -c "
import sys, json
d = json.load(sys.stdin)
print(d.get('tool_input', {}).get('file_path', ''))
" 2>/dev/null || true)

[[ "$file" == *.md ]] || exit 0
[[ "$file" == "$CLAUDE_PROJECT_DIR"/* ]] || exit 0

if ! result=$(markdownlint "$file" 2>&1); then
  echo "$result"
  exit 2
fi
```

Ключові рішення:

- `python3` для JSON-парсингу — `jq` не завжди встановлений.
- Ранній `exit 0` для не-`.md` файлів — хук не сповільнює роботу з іншими типами.
- Перевірка `$CLAUDE_PROJECT_DIR` — виключає файли поза проєктом (наприклад, памʼять агента в `~/.claude/`), щоб не застосовувати проєктний лінтинг до зовнішніх файлів.
- `exit 2` зі stdout помилок → агент бачить конкретні порушення і виправляє.

## Реєстрація в settings.json

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/md-lint.sh\""
          }
        ]
      }
    ]
  }
}
```

## Відома поведінка: блокування між едітами одного файлу

Якщо агент виправляє файл з кількома порушеннями за кілька `Edit`-викликів поспіль — хук спрацьовує після **кожного** і блокує, доки файл не стане повністю чистим.

Приклад: файл має MD040 у 3 блоках → агент виправляє перший → хук знаходить ще 2 → `exit 2` → агент продовжує → виправляє другий → хук знаходить 1 → `exit 2` → третій → `exit 0`.

Це коректна поведінка — агент отримує точний зворотний зв'язок після кожного кроку. Для файлів з багатьма порушеннями ефективніше переписати весь файл одним `Write`-викликом.
