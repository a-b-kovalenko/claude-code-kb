## 📝 TL;DR

Claude Code має вбудованих субагентів, які запускаються автоматично — без будь-якої конфігурації з вашого боку. Вони спрацьовують непомітно: `Explore` для пошуку файлів, `Plan` для збору контексту в plan mode, `general-purpose` для складних задач, що потребують і читання, і змін.

## Вбудовані субагенти

| Агент | Модель | Інструменти | Коли Claude його використовує |
| :--- | :--- | :--- | :--- |
| **Explore** | Haiku | Read-only (Write/Edit заборонені) | Пошук файлів, аналіз структури коду, дослідження без змін |
| **Plan** | Успадковує | Read-only (Write/Edit заборонені) | Збір контексту кодобази під час plan mode |
| **general-purpose** | Успадковує | Усі | Складні багатокрокові задачі: і дослідження, і зміни |
| **statusline-setup** | Sonnet | — | Виклик `/statusline` для налаштування рядка стану |
| **claude-code-guide** | Haiku | — | Питання про функціональність Claude Code |

> `statusline-setup` та `claude-code-guide` запускаються автоматично — викликати їх напряму не потрібно.

## Особливості Explore та Plan

На відміну від `general-purpose` та кастомних субагентів, **Explore** і **Plan** не завантажують:

- `CLAUDE.md` та `rules/` з проєкту
- поточний git status

Це зроблено навмисно: прискорює роботу та здешевлює кожен виклик, оскільки ці агенти займаються лише читанням і не потребують контексту правил.

## Thoroughness у Explore

При делегуванні до `Explore` Claude вказує рівень деталізації:

| Рівень | Коли |
| :--- | :--- |
| `quick` | Точковий пошук одного символу або файлу |
| `medium` | Помірне дослідження — знайти де щось визначено |
| `very thorough` | Глибокий аналіз кількох локацій та конвенцій |

## Чи конфліктує вбудований Explore з кастомним explorer?

**Конфлікту немає, але вони різні агенти з різними призначеннями.**

Ідентичність субагента визначається полем `name:` у frontmatter. Вбудований має ім'я `Explore`, кастомний зазвичай `name: explorer` — це різні ідентифікатори, жодного override за назвою не відбувається.

### Функціональна різниця

| | Вбудований `Explore` | Кастомний `explorer` |
| :--- | :--- | :--- |
| Знання проєкту | Загальне | Специфічне: архітектура, шари, Kafka flow |
| Читання файлів | Уривки, може пропустити поза вікном | Те саме, але знає де шукати |
| Підходить для | Знайти файл, grep символу, точковий lookup | Трасування залежностей між модулями, пошук споживачів Kafka-топіку, навігація по шарах |

**Кастомний `explorer` переважатиме для задач, пов'язаних з конкретним проєктом** — не тому що "перекриває за іменем", а тому що його `description` точніше збігається з задачею. Claude обирає агента за описом.

### Як вимкнути вбудований Explore

Вбудований `Explore` не перекривається кастомним автоматично. Єдиний задокументований спосіб його вимкнути — `permissions.deny`:

```json
{
  "permissions": {
    "deny": ["Agent(Explore)"]
  }
}
```

## Обмеження: субагенти не можуть породжувати субагентів

Субагент не може запустити іншого субагента. Це запобігає нескінченному вкладенню. Саме тому `Plan` існує окремо від `general-purpose`: коли ви вже всередині субагента, він не може делегувати дослідження далі — але вбудований `Plan` усе одно виконує збір контексту.
